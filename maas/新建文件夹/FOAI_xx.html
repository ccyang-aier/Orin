01 · 问题定位
易用性不是功能，是认知负荷管理
"门槛低"只是结果。真正的问题是：用户在完成一个目标前，需要在脑中维持多少概念、做多少决策、承受多少不确定性？
传统视角（特性清单）
✓ 提供 GUI 界面
✓ 接入 API Key 管理
✓ 内置监控看板
✓ 集成 AI Copilot 对话框
→ 人人都能罗列，缺乏区分度
认知负荷视角（本质洞察）
外在负荷：交互设计造成的无谓复杂
内在负荷：任务本身固有的复杂度
关联负荷：理解概念间关系的成本
→ 不同负荷需要不同策略应对
认知负荷理论（Cognitive Load Theory）源自 Sweller 1988，被广泛应用于 UX 设计评估框架

06 · L3 洞察（核心）
平台自举：FOAI 的结构性差异优势
FOAI 本身运行 LLM 推理服务——这意味着平台可以将推理能力内化为自身的"元能力"，实现自我优化、自我认知、自我进化。纯 IaaS/PaaS 不具备这一先天条件。
自举能力的三个表现
知识自生成
LLM Wiki：用模型生成关于模型的文档，运维知识库实时更新，无需人工维护
经验持久化
tCache → Agent 多级记忆：异常处理经验、用户偏好、操作模式沉淀为可复用知识
路由自优化
SmartModelHub：根据任务语义和历史效果自动匹配最优模型，形成闭环反馈
为何竞争对手难以复制
纯云厂商 IaaS：提供算力，无推理语义
纯 API 网关：转发请求，无上下文积累
通用 MLOps：管理模型，不运行推理

FOAI：既是推理服务的提供者，也是推理能力的使用者
这是一种飞轮效应的起点，而非单纯的功能叠加
自举（Bootstrapping）概念：系统用自身能力提升自身能力，形成正反馈

09 · 总结
四个核心判断
如果只记住一件事：易用性不是功能的堆叠，是推理平台独有的自我进化路径。
1 重新定义问题域
易用性 = 认知负荷管理，而非功能完备性。三类负荷对应三种截然不同的设计策略。
2 业界正发生范式转移
AI-Bolted 到 AI-Native 不是渐进演化，是架构决策的分叉点。现在是 FOAI 卡位的窗口期。
3 自举优势是真正的护城河
FOAI 运行推理服务 + 消费推理服务的双重身份，构成竞争对手无法轻易复制的结构性壁垒。
4 飞轮而非里程碑
路线图的终点不是"完成易用性改造"，而是启动一个数据 → 认知 → 体验 → 能力的自强化循环。
一句话定位：FOAI 易用性竞争力的终极形态，是一个用自身推理能力持续降低自身使用门槛的自进化系统。
