根因分析
为什么AI推理平台"难用"？
易用性的本质是消除不确定性——配置、行为、认知三重叠加，是所有使用障碍的共同根源。
①
配置不确定性
参数空间爆炸，用户无法预判合法配置边界，只能靠试错。
batch_size × concurrency × tensor_parallel 的组合空间难以穷举
镜像版本与驱动兼容性隐性依赖
服务上线后参数效果不可预期
②
行为不确定性
服务运行时"黑盒"——性能波动、排队积压、异常告警缺乏可解释反馈。
延迟抖动无法归因到具体资源瓶颈
告警只报结果，不给根因和处置建议
拓扑视图有，但运维决策信息缺失
③
认知不确定性
用户心智模型与平台模型错位——不知道"现在到底发生了什么"。
概念映射不一致（服务/实例/副本边界模糊）
操作路径分散，无统一心智地图
跨模块信息孤岛，需自行串联诊断链路
参考：Taming the Titans: A Survey of Efficient LLM Inference Serving（ACL 2025）——推理配置复杂度的系统性量化分析

L1 短板补课
L1阶段：已完成与待补齐项
用三重不确定性框架重新归因L1问题，而非简单的功能清单。
不确定性类型 典型问题 当前状态 补课方向
配置不确定性 参数组合无校验，部署失败反馈滞后 部分覆盖 前置参数合法性校验 + 冲突检测提示
配置不确定性 镜像兼容性隐性依赖，上线后才暴露 待完善 镜像-驱动-模型三元依赖预检
行为不确定性 部署进度"黑盒"，无阶段性状态反馈 待完善 部署全流程状态机可视化
行为不确定性 异常告警无根因归因，需人工排查 基础告警已有 结构化故障诊断（原因+影响+建议）
认知不确定性 API Key权限边界不清晰 已有基础管理 权限模型可视化说明
认知不确定性 模型评估与推理服务割裂，缺闭环 模块独立 评估→服务调优的操作连通
以上问题已通过AI4RD用户反馈数据归因，与三重不确定性框架高度吻合，非主观推断。
L2 可理解性 · 观测
让服务状态"自己开口说话"
从"有监控"到"会解释"——L2观测体系的核心是主动、可解释、面向决策。
SLO健康度看板
服务健康的统一语言
以SLO达标率作为单一健康指标替代原始metrics堆砌。用户不需要懂P99/QPS，只需看"健康/警告/危险"三态，并附带归因信息。
排队状态透出
消除"请求去哪了"的焦虑
显示当前队列深度、预估排队时长、资源瓶颈位置。与现有拓扑视图联动，点击指标直接定位问题节点。
拓扑运维增强
从"可视化"到"可操作"
在端到端拓扑节点上叠加健康状态、历史异常频次；拓扑节点支持直接触发扩缩容、重启等操作。

Azure AI Foundry
结构化诊断页：问题描述 → 影响范围 → 根因 → 推荐操作，四段式呈现，消除认知不确定性。

LLM在故障全链路（检测→归因→修复）的应用已有系统性实证，FOAI的结构化诊断方向与此高度吻合。
核心判断
可观测性不是监控的升级，而是用户认知负担的转移——从用户脑子里转移到系统里。
L2 可理解性 · 部署
部署与生命周期的认知简化
Golden Path不是限制用户选择，而是在正确的位置设置正确的默认值——降低的是认知负担，不是能力上限。
场景化部署模板（Golden Path）
为80%的场景提供开箱即用的起点
预设：高并发推理 / 低延迟对话 / 批量离线处理 / 多模态服务 四类模板，每类模板包含推荐参数组合及适用说明。用户从模板出发，按需调整，而非从空白页面开始。
参考：Platform Engineering社区Golden Path自服务LLM部署范式（2025）
前置参数校验
把错误消灭在部署之前
在提交前完成：参数合法性 × 资源充足性 × 镜像兼容性三层检查，给出结构化错误提示（原因 + 修复建议），而非部署失败后的日志排查。
生命周期结构化诊断
从模型上线到下线的全程状态可读
每个生命周期节点（注册→部署→服务→缩容→下线）均提供状态说明与上下文，用户在任意节点都清楚"当前在哪、下一步是什么"。
Golden Path概念来自DORA/Platform Engineering实践，已在Spotify、Google内部平台工程中得到大规模验证。

面向Agent的双面易用性
易用性的新维度：Agent Consumer Interface
FOAI不仅要对人友好，还要对Agent友好——ACI（Agent-Computer Interface）设计是MCP Server化的易用性前提。
ACI 设计原则 · 工具定义
工具边界清晰，无歧义
每个MCP Tool的名称、参数、返回值须从Agent决策视角设计，而非从内部实现视角暴露。工具粒度以"一次可完成的最小有意义操作"为单位。
ACI 设计原则 · 错误反馈
错误信息为Agent可重试而设计
Tool调用失败时，返回结构化错误：原因分类 + 可重试性标记 + 修正建议。Agent可据此自主决策下一步，而非依赖人工介入。
ACI 设计原则 · 上下文
减少Agent的上下文消耗
Tool返回值精简至Agent决策所需的最小信息集；避免返回冗余字段增加Agent的推理负担和context token消耗。
与架构师方向衔接
MCP Server化的易用性保障
MCP Server化是接口层变革，ACI设计是其质量标准。没有ACI设计规范，MCP Server只是又一个API wrapper，无法真正支撑Agent编排场景。
ACI设计原则参考：Anthropic "Writing Effective Tools for AI Agents"（2025）；OpenAI Agents SDK内置guardrails与可观测性设计思路

核心判断
三个不会过时的结论
01
易用性的竞争不在功能多少，而在不确定性消除的深度
竞品都在做AI Copilot、都在做自动化，真正的差距在于：谁能系统性地消除配置、行为、认知三重不确定性。这是能力，不是功能。
02
L2是当前的核心战场，L3是必须布局的方向
L1补课解决"能用"，L2可理解性建立"好用"的护城河，L3意图驱动是2026年以后差异化的唯一可能路径——越早布局，越难被追上。
03
易用性必须同时面向人和Agent，否则将在下一个周期掉队
Agent编排已是现实场景，ACI设计质量决定FOAI能否成为Agent生态的基础设施。MCP Server化是接口变革，ACI是其质量门槛——不能本末倒置。
