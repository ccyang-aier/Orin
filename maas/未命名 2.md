| 🟡 变更故障（版本 / 环境变更引发）    | 17  | **17%** | 框架升级致接口不兼容、服务中断     |

### 高频痛点 Top 5（用户原声）

**① 错误信息不可读** — 53+ 次收到 400/500 错误码，无法判断根因；用户明确需求："*希望报错可以明确具体原因*"

**② 任务堆积无感知** — 20+ 次反馈"模型卡顿 / 响应慢"，用户无法区分是模型问题还是排队问题

**③ 配置参数黑盒** — 上下文长度、Tool Call 参数、reasoning-parser 等关键参数，无引导、无提示、无校验，多次导致生产问题

**④ 告警信息混乱** — 告警 ID 重复、级别不合理、无法跳转到对应服务，排障效率低

**⑤ 跨模块信息割裂** — 各子系统无法互跳，概念术语不统一（GPU / 加速卡混用），集群信息展示不一致

## P8 · 模型全生命周期易用

**目标：覆盖模型管理、选型、评估、体验全链路，降低决策与操作成本**

### 全链路覆盖现状 & 断层分析

```
模型获取          →     模型管理          →     模型选型 & 评估
模型广场 ✅           模型列表 ✅               模型体验 ✅
                      版本管理 ✅               竞技场 ✅
                      量化管理 ✅               评估体系 ✅
                                                    │
                               ← ← ← ← 断层：评估结果与部署决策割裂
                                                    ▼
                                         推理服务部署 → 业务使用
```

| 断层点         | 现象                    |
| ----------- | --------------------- |
| 模型广场 → 部署   | 缺少"适合我场景"的引导，用户自行摸索   |
| 评估结果 → 部署决策 | 评估报告与部署流程割裂，无法直接转化为行动 |
| 使用中 → 问题反馈  | 遇到问题无法快速关联到模型配置，排查链路长 |
| 监控指标 → 模型调优 | 性能数据有，但对用户的调优建议缺失     |

### 演进规划（5项）

- 🎯 **模型选型引导**：在模型广场提供场景化推荐标签（代码生成 / 长文档 / 工具调用 / 多语言），帮助用户快速定位适合业务的模型
- 🎯 **评估 → 部署一键贯通**：评估结果页直接提供"以当前最优配置部署"入口，评估结论即时转化为行动
- 🎯 **模型健康度看板**：整合评估评分、运行时性能指标、用户反馈，提供统一的模型健康度评估视图
- 🎯 **Anthropic 接口支持（v1/messages）**：满足 Claude Code 等主流工具直接对接，消除高频兼容性投诉（用户需求：13条相关反馈）
- 🎯 **Token 消耗多维统计**：支持按部门、节点、时间段多维统计，历史数据可溯源（当前仅 14 天），满足成本管控与分摊诉求

## P9 · 交互一致性与认知易用

**目标：消除概念混淆与信息割裂，构建统一的平台心智模型，降低用户认知负担**

### 当前一致性问题清单

**术语不统一**
- "加速卡" vs "GPU"：监控与统计模块混用，同一硬件两种叫法
- "文件管理导入" vs "文件管理引入"：功能区别不明，用户困惑

**结构不一致（同一"个人/组织/平台"三层结构，各模块命名与顺序各不相同）**

| 模块   | 结构                        |
| ---- | ------------------------- |
| 镜像管理 | 我的镜像 / 组织的镜像 / 平台的镜像      |
| 文件管理 | 平台目录 / 我的目录 / 组织目录 / 用户目录 |
| 模型列表 | 我的模型 / 组织的模型 / 模型广场       |
| 训练模板 | 我的模板 / 平台模板               |

**跨模块割裂**
- 各子系统页面无法相互跳转，首页 TOP 点击无跳转
- 集群信息在首页与模型概览页展示字段、顺序不一致

### 演进规划（4项）

- 🎯 **统一术语规范**：建立平台统一词汇表，全局对齐 GPU / 加速卡、个人 / 我的 / 用户等术语，消除歧义
- 🎯 **统一资源层级结构**：全模块采用"个人 / 组织 / 平台"三层结构，命名与排列顺序保持一致，降低用户学习成本
- 🎯 **跨模块智能跳转**：关键数据点（首页 TOP 服务、告警主体、拓扑节点）支持直接跳转至对应服务详情，打通"看到问题→立即处理"链路
- 🎯 **API Endpoint 稳定化**：消除随机字符串 Endpoint（如 `/eb4c099d...`）引起的调用层变更困难，提供稳定可读的服务访问地址

> **核心主张**：一致性不是细节问题，是用户信任的基础。平台越大，一致性收益越显著——每消除一个认知障碍，就是为每一个用户节省一次排查时间。

## P10 · 异常感知与闭环处置易用

**目标：让错误可读、可追、可自愈，告警即行动指引，从"发现问题"到"解决问题"全程贯通**

### 问题现状：用户面对错误"两眼一抹黑"

**真实错误信息（用户高频收到）：**
```
{"msg": "Service call failed, call returned: 400 BAD_REQUEST", "code": 500, "data": null}
```
> 此类报错在 102 条反馈中出现 **15+ 次**，背后覆盖了上下文超限、格式不兼容、接口不支持等**完全不同的根因**，但用户收到的是**完全相同的报错**，无从判断、无从处理。

### 告警体系现存问题

| 问题                                     | 影响                  |
| -------------------------------------- | ------------------- |
| 告警 ID 重复，同一 ID 出现多条含义不同的告警             | 运维人员困惑，无法判断严重程度     |
| 告警描述缺少统计周期与具体卡信息                       | 无法快速定位是哪张卡、哪段时间出了问题 |
| 告警级别不合理（如 GPU 内存 row-remapping 失败被标低级） | 严重问题被忽视，响应延迟        |
| 告警主体无法跳转或全局搜索                          | 看到告警无法直接进入处置，行动链路断裂 |

### 演进规划（5项）

- 🎯 **错误信息结构化与可读化**：将平台错误分类（上下文超限 / 格式不兼容 / 服务过载 / 配置错误），提供人类可读的错误描述 + 具体原因 + 推荐操作
- 🎯 **告警体系重构**：告警 ID 唯一化；描述包含具体设备信息、统计周期、当前值 vs 阈值；级别与实际影响匹配；支持一键跳转故障服务
- 🎯 **日志保留与追溯**：历史日志从 14 天延长至 90 天，支持按服务、时间段、错误类型检索；根因追溯不因日志清除而断链
- 🎯 **排队状态主动透出**：推理服务排队时主动展示当前排队位置、预计等待时间，让"慢"有迹可循，减少无效重试与用户投诉
- 🎯 **僵尸任务自动清理 & QoS 控制**：自动识别并释放无效占用 GPU 的僵尸任务；支持大 Token 任务限流，保护核心业务 SLO

## P12 · 总结 · FusionOne AI 易用性演进全景

### 易用性建设核心主张

> **易用 = 降摩擦**
> 不是让平台更简单，而是在每个用户与平台发生交互的节点，主动消除障碍。

### 五大方向与用户旅程映射

```
用户触达平台
      │
  ② 部署易用              ← 降低入门门槛，零配置快速上线
      │ 服务运行中
  ① 观测与排障易用         ← 问题快速定位，分钟级锁定瓶颈
      │ 出现异常
  ⑤ 异常感知与故障闭环易用     ← 错误可读可操作，排障全程贯通
      │ 模型选型 & 决策
  ③ 模型全生命周期易用     ← 选型到评估到部署，链路零断层
      │ 日常平台操作
  ④ 交互一致性与认知易用   ← 消除认知障碍，构建信任基础
      │
  用户信任积累  →  竞争力壁垒形成
```
