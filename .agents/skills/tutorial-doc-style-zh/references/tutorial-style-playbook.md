# 教程写作 Playbook

本文件承载可按需加载的详细写作规则。主工作流和硬门控以 `SKILL.md` 为准；这里用于帮助代理设计更好的文章结构、读者体验、来源体系和学习测评。

## 1. 读者体验目标

优秀教程应当像长期学习资产，而不是聊天记录、执行日志、百科词条或图注拼贴。

目标体验：

- 开头要帮助读者进入学习语境，但具体切入方式由主题决定；
- 读者能快速理解主题为什么重要；
- 第一个概念钩子要具体，不要从抽象定义开始；
- 定义用于压缩理解，不要把定义当作所有教程的固定开场；
- 章节转场自然，文章像一个连续的教学过程；
- 图表服务于正文推理，正文不要退化成一串图片说明；
- 教学插图是一等资产：逻辑清晰、内容生动、视觉体验美观舒服的插图，能显著降低理解门槛，也更容易让读者愿意继续读下去；
- 机制解释需要把图表、公式、例子和来源支撑结合起来；
- 中心机制要有足够篇幅，不把必要推理、代价模型、约束或失败模式压成浅层旁白；
- 工程判断要明确：何时使用、何时不使用、会出什么问题、如何检查；
- 参考资料清晰可追溯，便于未来复查。

## 2. 教学结构工具箱

把这些当作教学动作，而不是固定大纲。根据主题的学习问题、读者基础和发布环境选择、合并、重排或省略。

在写作、改写、扩写、超出最小定位的研究、设计最终图示、生成图片、编辑目标 artifact 或运行质量门之前，必须先建立 teaching brief 并停止。只有用户明确确认或修订 brief 后才能继续。brief 前的最小定位仅限读取用户请求、仓库说明、目标路径、目标文档标题或短摘录，以及必要的系列文件名或索引。

- 目标读者及其先验知识；
- 内容边界、深度标准，以及明确排除的读者、场景或子领域；
- 文章必须回答的中心问题；
- 该文在更大学习路径中的独特教学承诺；
- 可图解知识点、视觉风格、图表密度和图表计划；
- 内容目录安排；
- 学习测评重点；
- Multi-Agent Quality Gate 的执行方式和必要授权。

常用结构动作如下。它们是可选教学动作，不是默认章节顺序；每篇教程都要根据已确认的 brief 重新组织。

1. 元数据：
   - 只在目标环境需要时添加 frontmatter、tags、updated、description 等字段；
   - 不要把编辑器、网站或仓库特定字段强塞进通用文档。
2. 标题：
   - 保留用户要求的标题，除非它明显只是临时占位；
   - 使用单一 H1。
3. 导入：
   - 用普通语言说明核心学习承诺；
   - 只有在目标平台支持且确实改善阅读体验时才使用 callout。
4. 语境与问题：
   - 给足进入问题的叙事 runway；
   - 避免一开头就跳进很窄的机制或字典定义；
   - 解释这个概念为什么被系统、工程实践或理论问题逼出来。
5. 定义：
   - 给出紧凑定义；
   - 在读者已有足够理解钩子时，用相邻概念对比减少误解。
6. 机制：
   - 分步骤解释核心机制；
   - 公式只在能澄清机制时使用；
   - 公式要配 prose 解释和小例子。
7. 深入：
   - 对结构性重要的依赖、成本、协调机制、约束或失效模式设置完整小节；
   - 讨论会影响真实决策的成本、边界、变体和操作后果。
8. 工程实践：
   - 展示概念在主流工具、命令、代码路径或部署模式中的样子；
   - 明确约束和 trade-off。
9. 误区：
   - 写出常见错误心智模型；
   - 用清晰区分纠正它们。
10. 总结：
   - 用一两段重建整篇文章的概念结构；
   - 优先给可复用心智模型，而不是口号。
11. 参考资料：
   - 默认使用一个有序列表，除非用户要求分类。
12. 学习测评：
   - substantial tutorials 必须在参考资料之后添加固定章节 `学习测评`；
   - 至少 10 道定制题，以单选题和多选题为主；
   - 先列出所有题目，再列答案和解析；
   - 覆盖核心概念、迁移场景、常见误区、困难机制和实现细节。

顶层教学章节控制在 6-7 个以内，不含参考资料与 `学习测评`。过多主章节会让教程碎片化，削弱文章地图。

标题要短、正式、耐久。标题是导航锚点，不是压缩段落；避免句子式标题、聊天式标题、依赖当前图号的标题，把解释放进正文。

## 3. 系列一致性与非公式化

系列教程要有连续性，但不能让前置上下文挤占当前任务的思考空间。先轻量定位系列结构，再按需要读取最少上下文。

写系列新篇章前：

- 先查看所在目录结构、索引、文件名序列或导航文档，理解系列的主题版图和当前位置；
- 不默认读取全量系列文章，避免任务开始前被历史上下文撑爆；
- 按主题选择叙事引擎，例如操作故事、概念对比、机制拆解、失败模式调查、决策树、历史演化、系统生命周期、源码路径、工程案例或心智模型阶梯等。

好模式：

- 保留标题风格、符号、图表质量和测评严谨度，同时让每个概念决定自己的结构，拒绝千篇一律的叙事模式。

## 4. 图文配合

详细视觉模式见 `visual-style-baselines.md`。这里关注图表如何服务教学。

教程应把 `imagegen` 教学图作为第一等资产，而不是正文写完后的装饰。凡是可以通过图片讲清或辅助讲清的概念、流程、结构、对比、状态变化、边界条件、决策路径，都应主动规划插图。图多不是目的；目标是让每张图都能让读者更愿意读、更容易懂、更容易记住。

适合画图的对象：

- 分布式架构；
- tensor/data flow；
- before/after 对比；
- memory layout；
- communication timeline；
- system topology；
- conceptual map；
- lifecycle diagram。

图文规则：

- 每张图都要有附近阅读指导，但形式可以变化：短段落、图注、紧凑 note 或上下文解释；
- 避免每张图都套同一个“图 X 读图顺序”模板；
- 正文驱动学习路径，图表支撑正文，不反客为主；
- 长技术教程中，3 张图通常偏少，除非文章本身较短；
- 每张图必须有一个清晰教学目的；
- 讲 computation、data flow、state flow、coordination 的图必须展示足够结构，使机制可检查；
- 中文教程中，读者辅助标签优先使用中文；论文、代码、API、模型卡、框架日志中的稳定专业词保留英文；
- 避免中文教程中的教学图全英文，除非是截图或外部源图；
- 讲实现机制时，要明确 before/after 状态、所有权、数据布局、通信边界和约束；
- collective operation 或分布式协调相关图要展示操作后的状态，不只是写出操作名；
- 生成图里的文字也是技术内容，不是装饰；
- 追求 paper-level clarity：清晰布局、可读标签、准确箭头、克制颜色、无装饰噪声。

## 5. 来源标准

技术教程写作前先建立证据。

优先组合：

- 原始论文或 canonical paper；
- 官方框架文档；
- GitHub 仓库、源码文档或设计文档；
- 高质量社区教程或长文解释。

不要混淆来源边界：

- “论文提出/证明”不等于“框架当前支持”；
- “官方文档建议”不等于“社区通常这么做”；
- “本地解释”要写成解释，不要伪装成被引用事实。

对容易漂移的事实，优先使用当前官方资料，并记录稳定链接。

## 6. 语气与深度

推荐：

- 短概念段落；
- 新问题前有语境铺垫；
- 用具体例子降低抽象负担；
- 公式引入新代数区别时给小 toy example；
- 对比表只在能强化区分时使用；
- 精确术语配中文解释；
- 使用“为什么这个设计存在”的框架；
- 对成本模型、变体、约束、操作检查等核心知识点进行深入处理。

避免：

- hype language；
- 没有机制支撑的空泛形容词；
- 滥用引号强调；
- 随意或一次性标题；
- 重复图注模板；
- LaTeX 公式周围无意义空行；
- 过长历史离题；
- 过度 H4/H5 嵌套；
- 装饰性分割线；
- 无来源且未限定的不确定断言；
- 写入对话过程或代理执行过程。

## 7. 学习测评标准

学习测评是教程质量的一部分，不是附录填充。

- 题目要测试真实理解，不只测试记忆；
- 错误选项要来自可信误区、相邻策略或细微陷阱；
- 不使用荒谬、无关或玩笑式干扰项；
- 场景题选项要具体且技术上可判定；
- 混合基础检查、迁移场景和深层推理；
- 多个选项看起来都合理时，解析要更详细；
- 长测评应提供复盘路径；
- 题目完成后必须运行 assessment reviewer。

## 8. 长期偏好

- 目标文档家族期待编号标题时，教程笔记使用编号标题；
- 长期笔记必须脱离当前对话也能自洽阅读；
- 优先使用 source-backed explanation，避免无依据自信；
- 空文档或 outline-like 文档应扩展为完整 artifact，而不是另建旁支文档；
- AI-era 教程把高质量图表视为一等教学资产；
- 图表引入陌生对象、中间状态、实现角色或领域操作时，必须配附近解释；
- 能用图片讲解或辅助讲解的地方，应尽可能规划 `imagegen` 教学插图，但每张图都要承担明确教学任务；
- 双语标签要有意图：中文降低理解成本，英文保留读者会在论文、文档和代码中看到的标准术语；
- 分布式、并行、多组件系统教程要说明协调发生在哪里，边界跨越后有什么状态或布局，代价或可靠性含义是什么；
- 工程向主题至少连接一个真实部署或使用场景；
- reader-experience review 必须判断叙事连续性、铺垫、转场，以及文章是否像完整教程而不是图解集合；
- 生成图的事实标签要像正文一样审查，尤其是数据所有权、状态转换、算子、协议和框架关系；
- handdrawn 图优先像认真教学板书：人为构图、可读标签、一致语义颜色、充足留白、无通用 sketch 特效。

当用户给出新的纠正时，只沉淀可复用偏好，不加入一次性任务细节。
