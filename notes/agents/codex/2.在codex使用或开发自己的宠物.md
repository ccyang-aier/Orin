# 在 Codex 使用或开发自己的宠物

这篇笔记记录 Codex Pets 的使用、安装和自定义开发方式。需要先说明一个边界：截至 2026-05-10，我能检索到的 OpenAI 官方资料主要说明 Codex 是用于写代码、审查代码、交付代码的 AI coding agent；关于 Codex 宠物的包格式、安装路径、动画网格和社区下载站，主要来自社区站点与本地 `hatch-pet` 技能说明，不应直接当作 OpenAI 官方接口承诺。

## Codex 宠物是什么

Codex 宠物可以理解为 Codex app 里的一个小型动态陪伴组件。社区文档通常把它描述为会出现在 Codex 工作流旁边的 floating overlay：在任务运行、等待、失败、review 等状态下，宠物会切换不同动画状态，让长期工作会话有一点“桌面电子宠物”的反馈感。

它不是 Codex 的核心编码能力，也不影响模型本身的推理、工具调用或代码修改。它更像是一个轻量 UI/情绪反馈层：有趣，但别把它和 agent 能力、模型版本、自动化能力混在一起。

## 已知功能与交互

社区资料与当前 Codex app 的常见用法里，宠物相关入口主要有两个：

- `Settings -> Appearance -> Pet`：选择内置宠物或添加/选择自定义宠物。
- `/pet`：在 Codex composer 里唤出或收起当前宠物。

宠物通常会覆盖这些状态：

- `idle`：空闲状态，适合轻微呼吸、眨眼或小幅晃动。
- `running-right`：向右移动或拖拽时的运动状态。
- `running-left`：向左移动或拖拽时的运动状态。
- `waving`：挥手、打招呼。
- `jumping`：跳跃。
- `failed`：失败、阻塞或出错反馈。
- `waiting`：等待用户输入或等待下一步。
- `running`：任务执行中的工作状态，不一定是字面意义的奔跑。
- `review`：等待审查、检查结果或 review 阶段。

这些状态名称来自社区安装指南和本地 `hatch-pet` 技能的动画行约定。实际 UI 呈现可能随 Codex app 版本变化而改变。

## 内置宠物与社区宠物

CodexPets.net 资料提到，内置宠物可在 Codex 的 `Settings -> Appearance -> Pets` 中选择，自定义宠物可以以下载包形式添加。`hatch-pet` 技能说明里也列出了一些内置风格示例名称，例如 `Codex`、`Dewey`、`Fireball`、`Rocky`、`Seedy`、`Stacky`、`BSOD`、`Null Signal`。

社区宠物通常来自这些类型的站点：

- `CodexPets.net`：社区宠物浏览、下载与 DIY 入口；站点声明自己不是 OpenAI 官方资源。
- `CodexPets.org` / `codex-pet.org`：安装说明、包格式说明、生成教程、宠物目录。
- `Petdex`：面向 Codex 动态宠物的开源索引/图库，提供 CLI 安装和提交思路。

使用社区宠物时要注意版权与来源安全：不要随便导入明显侵犯 IP、包含可疑脚本或来源不明的包。一个正常宠物包一般只需要图片与 manifest，不应该要求你执行复杂脚本或提供敏感凭据。

## 宠物存储位置与包结构

社区安装指南给出的本地安装约定是：

```text
~/.codex/pets/<pet-id>/
  pet.json
  spritesheet.webp
```

在 Windows 上，`~` 通常对应用户目录，例如：

```text
C:\Users\<你的用户名>\.codex\pets\<pet-id>\
  pet.json
  spritesheet.webp
```

最小 `pet.json` 结构通常类似：

```json
{
  "id": "my-pet",
  "displayName": "My Pet",
  "description": "A concise pet description.",
  "spritesheetPath": "spritesheet.webp"
}
```

字段含义：

- `id`：宠物的稳定标识，建议使用小写英文、数字和短横线。
- `displayName`：在 UI 中展示的名称。
- `description`：一句话说明宠物风格或来源。
- `spritesheetPath`：指向同一目录下的 spritesheet 文件，通常是 `spritesheet.webp`。

如果导入后 Codex 里看不到宠物，优先检查三件事：目录名是否清晰独立、`pet.json` 是否能被解析、`spritesheetPath` 是否真的指向同目录下存在的图片文件。

## spritesheet 规格

社区资料和 `hatch-pet` 技能的共同约定是：

```text
总尺寸：1536 x 1872 px
网格：8 列 x 9 行
单帧：192 x 208 px
总帧数：72
格式：WebP 或 PNG，推荐透明背景
```

9 行通常对应 9 个状态：

```text
1. idle
2. running-right
3. running-left
4. waving
5. jumping
6. failed
7. waiting
8. running
9. review
```

每行 8 帧，按从左到右播放。制作时最容易出问题的地方不是 manifest，而是 spritesheet：角色跨格、背景没有透明、某一帧裁切、行顺序错位、帧尺寸不一致，都会导致预览异常。

## 下载第三方宠物

以社区站点下载宠物为例，流程通常是：

1. 打开宠物站点，例如 `https://codexpets.net`，选择一个宠物。
2. 下载 `.zip` 或 `.codex-pet.zip` 包。
3. 解压到独立目录，确认里面至少有 `pet.json` 和 `spritesheet.webp` 或等价 spritesheet 文件。
4. 把目录放到 `~/.codex/pets/<pet-id>/`，或在 Codex 的 `Settings -> Appearance -> Pet(s)` 中添加解压后的包。
5. 在 Codex 里使用 `/pet`，或者回到设置页选择该宠物。

Windows 手动安装示例：

```powershell
$petId = "my-pet"
New-Item -ItemType Directory -Force "$env:USERPROFILE\.codex\pets\$petId"
Expand-Archive ".\my-pet.codex-pet.zip" -DestinationPath "$env:USERPROFILE\.codex\pets\$petId" -Force
```

安装后如果没有出现，可以重启或刷新 Codex，再检查目录是否变成了多包一层。例如错误结构可能是：

```text
~/.codex/pets/my-pet/my-pet/pet.json
```

理想结构应该是：

```text
~/.codex/pets/my-pet/pet.json
~/.codex/pets/my-pet/spritesheet.webp
```

## DIY 一个自己的 Codex 宠物

自己做宠物有两条路线：

- 手工路线：自己画 9 行 x 8 帧 spritesheet，再写 `pet.json`。
- 技能路线：使用 Codex 的 `hatch-pet` skill 生成基础角色、9 个动画行、QA contact sheet、预览视频和最终包。

如果目标是快速做一个能用的宠物，优先走 `hatch-pet`。它会把“生成图片”和“拼接成 Codex 可用 spritesheet”分开：视觉由 image generation 完成，确定性脚本负责切帧、验证、打包。

### 使用 hatch-pet 的推荐流程

在 Codex 中可以这样描述需求：

```text
Use hatch-pet to make a custom Codex pet: a small orange tabby cat named Miso, sleepy eyes, soft pixel style.
```

如果有参考图，可以附上图片并说明哪些特征必须保留：

```text
Use hatch-pet to make a Codex pet from this reference image. Keep the round ears, green scarf, sleepy expression, and compact chibi proportions. Name it Miso.
```

`hatch-pet` 的典型过程是：

1. 确定宠物名称、描述、参考图和工作目录。
2. 生成基础形象，作为后续所有动画行的身份锚点。
3. 生成 9 个状态行，保持同一个角色身份。
4. 生成最终 `spritesheet.webp`、`pet.json`、QA contact sheet、validation 文件和预览视频。
5. 把最终包写入 `${CODEX_HOME:-$HOME/.codex}/pets/<pet-name>/`。

需要特别注意：`hatch-pet` 不是一个已经安装好的宠物包，而是生成宠物的工作流技能。下载第三方宠物时不需要它；只有你要新建、修复或验证自定义宠物时才需要它。

### 设计提示词建议

好的宠物提示词应该短、具体、稳定，重点描述能在 192 x 208 小格子里看清的特征：

```text
A tiny Codex-style digital pet named Miso: compact orange tabby cat, sleepy green eyes, rounded head, tiny paws, green scarf, thick dark outline, limited warm palette, transparent-background sprite style.
```

建议明确这些要素：

- 物种或角色类型：猫、鸭子、机器人、石头、火球等。
- 轮廓：圆、矮胖、细长、有耳朵、有尾巴。
- 颜色：主色、点缀色、是否有围巾/帽子/小道具。
- 表情：困、兴奋、冷静、认真、捣蛋。
- 风格：pixel-art-adjacent、chibi、thick outline、limited palette。

尽量避免这些内容：

- 高细节写实毛发、复杂渐变、3D 质感。
- 大量文字、代码、UI 面板、Logo。
- 分离的特效，例如飘在旁边的星星、速度线、灰尘、阴影。
- 过细的小配饰，因为缩到 192 x 208 后会糊成噪点。

## 手工制作检查清单

如果不用 `hatch-pet`，至少检查这些项目：

- `pet.json` 是合法 JSON，没有尾逗号。
- `id` 与目录名语义一致，避免空格和中文路径问题。
- `spritesheetPath` 指向本目录里的真实文件。
- spritesheet 是 `1536 x 1872`。
- 每格是 `192 x 208`，一共 `8 x 9`。
- 每一行状态顺序正确。
- 背景透明，没有白底、黑底或棋盘格被画进图片。
- 角色没有跨格、裁切、重复错位。
- `running-left` 如果由 `running-right` 镜像得到，确认没有文字、单侧标记、单侧道具被错误翻转。

## 常见问题

### 安装宠物后没有出现怎么办？

先检查目录层级、manifest 和 spritesheet 路径。常见问题是 zip 解压后多了一层目录，导致 Codex 没有在预期位置看到 `pet.json`。

### `spritesheetPath` 可以是网络 URL 吗？

社区安装指南建议本地安装时指向宠物目录里的本地图片文件。某些网页预览器可能能加载 URL，但本地 Codex 使用时优先用相对文件名，例如 `spritesheet.webp`。

### 可以只做一个静态图吗？

理论上宠物包依赖 spritesheet 动画网格。只做一个静态图通常不够，除非你把同一形象复制到所有必要帧里，但体验会比较僵硬，也更容易和状态反馈脱节。

### 可以用自己喜欢的动漫/名人/IP 角色吗？

技术上社区里确实存在各种灵感来源的宠物，但长期使用和分享时要考虑版权、肖像权和平台规则。更稳妥的做法是做“原创但带某种气质”的角色，而不是直接复刻受保护角色。
