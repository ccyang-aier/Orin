# Codex 从入门到进阶：面向真实开发场景的完整使用指南

## 第 1 章：Codex 是什么，适合解决什么问题

### 1.1 Codex 的定位

Codex 是 OpenAI 提供的 **coding agent**，也就是面向软件开发任务的智能代理。它不是传统意义上的“代码补全工具”，也不只是一个能回答编程问题的聊天机器人，而是一个可以进入真实代码项目、理解上下文、读取文件、修改代码、运行命令、生成 diff、协助创建 PR、参与代码审查和 CI/CD 工作流的开发助手。

如果用一句话概括：

> Codex 是一个可以在你的真实开发环境中执行工程任务的 AI 编程代理。

这意味着 Codex 的使用方式和普通问答式 AI 有明显不同。你不只是问它“这段代码是什么意思”，还可以让它直接进入项目，完成类似下面这样的任务：

```text
请理解这个项目的整体架构，并告诉我主要模块分别负责什么。
```

```text
请定位登录失败的问题，做一个最小修复，并运行相关测试验证。
```

```text
请为这个组件补充单元测试，覆盖正常状态、加载状态和错误状态。
```

```text
请检查这个 PR 是否存在明显 bug、性能问题或安全风险。
```

在官方文档中，Codex 被描述为可以在多个开发入口中工作的统一代理，包括：

- Codex App；
- IDE Extension；
- Codex CLI；
- Codex Cloud；
- GitHub；
- Slack；
- CI/CD；
- SDK 和 Agents SDK。

也就是说，Codex 的目标不是只存在于某一个编辑器或某一个网页中，而是成为贯穿软件开发流程的智能协作层。

---

### 1.2 Codex 与 ChatGPT、代码补全工具、普通 IDE 插件的区别

很多人第一次接触 Codex 时，容易把它理解成“更会写代码的 ChatGPT”或者“更强的代码补全插件”。这种理解并不完全准确。

Codex 与其他工具的核心区别在于：它更强调 **任务执行**，而不仅仅是 **代码生成**。

| 工具类型 | 核心能力 | 典型使用方式 | 局限 |
|---|---|---|---|
| ChatGPT | 对话、解释、生成代码片段 | 复制代码或描述问题，然后询问 | 通常不直接进入项目上下文，不自动执行命令 |
| 代码补全工具 | 根据当前位置补全代码 | 在编辑器中写代码时自动建议 | 更偏局部补全，难以承担完整工程任务 |
| 普通 IDE 插件 | 辅助编辑、搜索、重构 | 在 IDE 内完成局部开发动作 | 多数不具备完整 agent loop |
| Codex | 理解项目、执行任务、修改代码、运行命令、生成 diff、参与协作 | 给出任务目标，让 Codex 探索、计划、修改、验证 | 仍需要人工审查、安全控制和测试验证 |

例如，当你使用普通聊天机器人时，可能需要这样工作：

1. 把错误信息复制给 AI；
2. 把相关代码片段复制给 AI；
3. 等 AI 给出建议；
4. 自己回到项目中修改；
5. 自己运行测试；
6. 如果失败，再把新错误复制回去。

而使用 Codex 时，这个流程可以变成：

```text
这个项目的登录流程在 token 过期后会失败。请复现问题，定位原因，做一个最小修复，并运行相关测试验证。
```

Codex 可以根据项目上下文自己查找相关文件、分析调用链、修改代码、运行命令，并向你汇报它做了什么。你仍然需要 review 它的改动，但它承担了更多“工程执行”的工作。

这也是为什么理解 Codex 时，最重要的不是把它看成一个“代码生成器”，而是把它看成一个“可以协作完成开发任务的 agent”。

### 1.3 Codex 的核心能力

从官方文档和使用入口来看，Codex 的能力可以概括为以下几类。

#### 1. 理解代码库

Codex 可以读取项目文件，理解目录结构、模块关系、技术栈、关键入口和运行方式。

适合的任务包括：

```text
请告诉我这个项目是做什么的，主要目录分别有什么作用。
```

```text
请解释用户登录请求从前端到后端的完整调用链。
```

```text
请帮我找出这个项目中最值得优先阅读的 10 个文件，并说明原因。
```

这类任务非常适合接手陌生项目、阅读大型代码库或快速做技术调研。

#### 2. 修改代码

Codex 可以根据目标直接修改项目文件。它可以完成小功能开发、bug 修复、代码重构、测试补充、文档更新等任务。

例如：

```text
请给设置页面增加一个深色模式开关，保持现有代码风格，不要引入新的依赖。
```

```text
请修复用户头像上传失败的问题，尽量做最小改动，并补充回归测试。
```

```text
请把这个复杂函数拆分成更小的函数，但不要改变外部行为。
```

#### 3. 运行命令和验证结果

Codex 不只是写代码，还可以运行项目中的命令，例如测试、类型检查、lint、构建等。

常见验证任务包括：

```text
修改完成后，请运行相关单元测试，并总结测试结果。
```

```text
请先运行 npm test 复现失败，再修复问题。
```

```text
请运行 typecheck 和 lint，确保没有新增错误。
```

这点非常关键。因为在真实开发中，代码是否“看起来合理”并不够，必须通过测试和构建验证。

#### 4. 生成 diff 和协助 review

Codex 的工作结果通常会以 diff 的形式呈现，开发者可以审查它修改了哪些文件、为什么修改、是否符合预期。

在 GitHub 集成场景中，Codex 还可以参与 PR review，例如检查：

- 是否存在明显 bug；
- 是否破坏现有行为；
- 是否存在安全风险；
- 是否有性能问题；
- 是否缺少测试；
- 是否违反项目规范。

#### 5. 参与团队协作和自动化流程

Codex 不仅可以在个人本地环境中使用，也可以进入团队开发流程，例如：

- 在 GitHub PR 中触发代码审查；
- 在 CI 失败后协助分析和修复；
- 在 Slack 中通过 `@Codex` 委托任务；
- 通过 GitHub Actions 把 Codex 放入自动化流水线；
- 通过 SDK 集成进企业内部工具。

这说明 Codex 的价值不只在“帮我写几行代码”，而是可能改变团队处理工程任务的方式。

### 1.4 Codex 适合解决什么问题

Codex 最适合的问题通常有三个特点：

1. **需要理解上下文**：不是孤立代码片段，而是要看项目结构、调用链、配置和测试。
2. **可以拆成明确任务**：目标、范围和验证方式相对清楚。
3. **结果可以被审查和验证**：可以通过 diff、测试、lint、typecheck、人工 review 来判断是否正确。

下面是几个典型场景。

#### 场景一：理解陌生代码库

当你刚接手一个项目时，可以先让 Codex 做代码导览。

示例 prompt：

```text
请先不要修改任何文件。请阅读这个项目，告诉我：
1. 这个项目的主要功能是什么；
2. 使用了哪些技术栈；
3. 主要目录和关键文件分别是什么；
4. 本地如何启动和测试；
5. 如果我要修改登录功能，应该先看哪些文件。
```

这种任务风险低、收益高，适合作为新项目的第一步。

#### 场景二：修复 bug

Codex 适合处理那些可以被复现、可以被测试、范围相对明确的 bug。

示例 prompt：

```text
用户在 token 过期后重新登录会失败。请你：
1. 找到相关登录和 token 刷新逻辑；
2. 分析可能原因；
3. 做一个最小修复；
4. 补充或更新相关测试；
5. 运行测试并总结结果。
```

这里的关键是不要只说“修复 bug”，而要说明现象、期望行为和验证方式。

#### 场景三：添加小功能

Codex 很适合做边界清楚的小功能。

示例 prompt：

```text
请在设置页面增加一个“导出数据”按钮。要求：
1. 沿用现有 UI 组件和样式；
2. 点击后调用已有 exportUserData 方法；
3. 成功时显示 toast；
4. 失败时显示错误提示；
5. 不要引入新的依赖。
```

这样的任务目标明确，约束清晰，更容易得到稳定结果。

#### 场景四：补充测试

很多项目缺少测试，而补测试往往需要理解现有代码结构。Codex 可以承担初步测试编写工作。

示例 prompt：

```text
请为 UserProfile 组件补充测试，覆盖：
1. 正常渲染用户信息；
2. 加载状态；
3. 接口失败状态；
4. 点击编辑按钮后的行为。
请使用项目现有测试框架，不要新增依赖。
```

#### 场景五：代码重构

Codex 可以做受控重构，但前提是必须明确“不改变外部行为”。

示例 prompt：

```text
请重构 src/services/payment.ts，让代码更易读，但不要改变外部 API 和业务行为。重构后请运行相关测试，并说明你做了哪些结构调整。
```

#### 场景六：代码审查

Codex 可以协助检查 PR，尤其适合发现一些常见问题。

示例 prompt：

```text
请 review 当前分支相对于 main 的改动，重点检查：
1. 是否有明显 bug；
2. 是否有安全风险；
3. 是否有性能问题；
4. 是否缺少测试；
5. 是否违反项目代码规范。
请按严重程度排序输出。
```

#### 场景七：CI/CD 故障修复

在 GitHub Actions 等 CI/CD 场景中，Codex 可以分析失败日志，定位问题，并尝试修复。

示例 prompt：

```text
CI 在 test 阶段失败。请阅读失败日志，定位原因，做最小修复，并确保相关测试通过。
```

### 1.5 Codex 不适合什么问题

Codex 很强，但并不意味着它适合所有任务。下面这些场景需要谨慎。

#### 1. 需求非常模糊的问题

例如：

```text
帮我优化一下这个项目。
```

这个任务太宽泛。Codex 不知道你想优化性能、架构、可读性、包体积、测试覆盖率，还是用户体验。

更好的写法是：

```text
请分析首页加载性能，重点关注首屏渲染时间、重复请求和大体积依赖。先输出分析报告，不要修改文件。
```

#### 2. 无法验证正确性的任务

如果任务没有测试、没有明确期望、没有可观察结果，Codex 就更容易“看起来完成了”，但实际上不一定正确。

例如：

```text
把系统改得更稳定。
```

这类任务需要先拆解成可验证目标。

#### 3. 高风险大范围改动

比如一次性要求 Codex 重写整个核心模块、迁移整套框架、重构所有接口。这类任务不是不能做，但不应该一次完成，而应该拆成多个小步骤。

更好的方式是：

1. 先让 Codex 分析现状；
2. 再提出迁移计划；
3. 人工确认方案；
4. 分模块执行；
5. 每一步都运行测试和 review diff。

#### 4. 涉及敏感权限和生产数据的任务

如果任务涉及生产数据库、密钥、用户隐私数据、支付系统或高权限操作，必须设置清晰边界，并由人工确认。

例如，不应该随意让 Codex 执行：

```text
连接生产数据库并修复里面的数据。
```

更安全的做法是让它先生成分析脚本、迁移草案或只读查询，然后由人工审核。

### 1.6 使用 Codex 的正确心智模型

使用 Codex 最重要的心智模型是：

> 把 Codex 当成一个有执行力的初级到中级工程协作者，而不是一个永远正确的自动开发机器。

这意味着你应该：

- 给它清晰任务，而不是模糊愿望；
- 给它上下文，而不是让它猜；
- 给它约束，而不是让它自由发挥；
- 要求它验证，而不是只生成代码；
- 审查它的 diff，而不是盲目接受；
- 用 Git、测试和 CI 给它兜底。

一个高质量 Codex 工作流通常是：

```text
明确目标 → 让 Codex 探索 → 让 Codex 制定计划 → 人工确认 → Codex 修改 → 运行测试 → 审查 diff → 合并或回滚
```

如果你按照这个流程使用 Codex，它就不只是“帮你写代码”，而是可以成为你开发流程中的稳定协作者。

## 第 2 章：四种使用入口：App、IDE、CLI、Cloud

Codex 官方提供了多种使用入口。不同入口适合不同开发习惯和任务场景。

如果你是第一次使用 Codex，建议从 **Codex App** 或 **IDE Extension** 开始；如果你习惯终端工作流，可以直接使用 **Codex CLI**；如果你希望让 Codex 在云端处理 GitHub 仓库任务，则可以使用 **Codex Cloud**。

本章会依次介绍这四种入口，并给出选择建议。

---

### 2.1 Codex App：最推荐的新手入口

Codex App 是官方提供的桌面端使用方式，适合想要快速体验 Codex 能力的新手用户。它的优势是入口直观、操作简单，并且适合在本地项目中启动多个 Codex 任务线程。

#### 适合人群

Codex App 适合：

- 第一次接触 Codex 的用户；
- 希望用图形界面操作的开发者；
- 想让 Codex 直接理解本地项目的人；
- 不想一开始就配置复杂命令行参数的人；
- 希望并行处理多个开发任务的人。

#### 基本使用流程

使用 Codex App 的基本流程如下：

1. 下载并安装 Codex App；
2. 打开 App；
3. 登录 ChatGPT 账号或配置 OpenAI API key；
4. 选择一个本地项目文件夹；
5. 选择 Local 模式；
6. 输入你的第一个任务。

你可以先从低风险任务开始，例如：

```text
Tell me about this project.
```

或者中文表达：

```text
请先不要修改任何文件。请阅读这个项目，并向我解释项目的用途、技术栈、目录结构、启动方式和测试方式。
```

这类任务不会修改代码，适合用来观察 Codex 是否能正确理解项目。

#### 第一个推荐任务：理解项目

对于初学者，不建议第一次就让 Codex 做大规模修改。更好的第一步是让它阅读项目。

推荐 prompt：

```text
请先不要修改文件。请阅读当前项目，并回答：
1. 这个项目的主要功能是什么；
2. 使用了哪些主要技术栈；
3. 核心目录分别有什么作用；
4. 本地如何启动；
5. 如何运行测试；
6. 如果我要修改登录功能，应该先看哪些文件。
```

这个任务可以帮助你验证三件事：

1. Codex 是否能读取项目；
2. Codex 是否能理解项目结构；
3. Codex 是否能给出有用的导航建议。

#### 第二个推荐任务：做一个小改动

当你确认 Codex 能理解项目后，可以让它做一个范围很小的修改。

例如：

```text
请在 README 中补充本地开发启动说明。请根据 package.json 中已有 scripts 编写，不要猜测不存在的命令。
```

或者：

```text
请修复页面中的一个明显拼写错误。只修改必要文件，并总结 diff。
```

这类任务风险低，容易审查，非常适合建立使用信心。

#### 使用 Codex App 的注意事项

使用 Codex App 时，要特别注意以下几点：

- 在任务开始前确保项目已经纳入 Git 管理；
- 重要任务前先创建 Git checkpoint；
- 第一次任务尽量选择“只读分析”；
- 不要一开始就让 Codex 大范围重构；
- 每次任务结束后都要看 diff；
- 对涉及依赖安装、删除文件、修改配置的操作保持谨慎。

### 2.2 IDE Extension：在编辑器中使用 Codex

IDE Extension 适合希望在日常编码环境中直接使用 Codex 的开发者。它通常嵌入在编辑器侧边栏或面板中，让你可以一边查看代码，一边让 Codex 执行任务。

#### 适合人群

IDE Extension 适合：

- 日常主要在 IDE 或代码编辑器中工作的开发者；
- 希望边看代码边和 Codex 交互的人；
- 经常做局部修改、测试补充、bug 修复的人；
- 希望把 Codex 融入日常开发流程的人。

#### 基本使用流程

使用 IDE Extension 的一般流程是：

1. 安装 Codex IDE 扩展；
2. 打开你的项目；
3. 打开 Codex 面板；
4. 登录 ChatGPT 账号或配置 OpenAI API key；
5. 确认当前工作目录；
6. 输入任务。

官方说明中，IDE Extension 默认以 **Agent mode** 启动。这意味着 Codex 不只是回答问题，而是可以读取文件、运行命令、修改当前项目目录中的代码。

#### IDE 中适合做什么任务

IDE Extension 非常适合中小范围开发任务，例如：

```text
请解释当前打开文件的核心逻辑，并指出它依赖了哪些模块。
```

```text
请为当前组件补充测试，使用项目现有测试框架，不要新增依赖。
```

```text
请检查当前分支改动，找出可能的 bug 和缺失测试。
```

```text
请把这个函数拆分得更清晰，但不要改变外部行为。
```

#### 为什么要使用 Git checkpoints

因为 Codex 在 Agent mode 下可以修改代码，所以建议你在重要任务前后创建 Git checkpoints。

简单理解，checkpoint 就是给当前代码状态打一个快照。如果 Codex 的修改不符合预期，你可以快速回退。

推荐工作流：

```text
确认当前代码状态 → 创建 checkpoint → 让 Codex 执行任务 → 查看 diff → 运行测试 → 接受或回滚
```

如果你的项目已经使用 Git，也可以通过普通 Git 命令完成类似操作：

```bash
git status
git add .
git commit -m "checkpoint before codex task"
```

或者如果你不想提交，也至少要确保工作区是干净的：

```bash
git status
```

这样你可以清楚地区分哪些改动是 Codex 产生的。

#### IDE Extension 的使用建议

在 IDE 中使用 Codex 时，建议遵循以下原则：

- 每次只给一个明确任务；
- 如果任务复杂，先让 Codex 输出计划；
- 明确说明可以修改哪些文件；
- 要求它使用项目已有风格；
- 修改后查看 diff；
- 运行相关测试；
- 不要盲目接受所有改动。

### 2.3 Codex CLI：在终端中使用 Codex

Codex CLI 是面向终端用户的入口，适合习惯命令行工作流的开发者。它可以在当前目录中启动 Codex，让 Codex 读取项目、执行任务、修改文件和运行命令。

#### 适合人群

Codex CLI 适合：

- 终端重度用户；
- 后端工程师、基础设施工程师、DevOps 工程师；
- 喜欢脚本化和自动化工作流的人；
- 希望在服务器、容器或远程开发环境中使用 Codex 的人；
- 希望把 Codex 接入更复杂流程的人。

#### 安装方式

官方 Quickstart 中提供了两种常见安装方式。

使用 npm 安装：

```bash
npm install -g @openai/codex
```

或者使用 Homebrew 安装：

```bash
brew install codex
```

安装完成后，可以在项目目录中运行：

```bash
codex
```

然后根据提示登录 ChatGPT 账号或配置 OpenAI API key。

#### 基本使用流程

CLI 的典型使用流程是：

1. 打开终端；
2. 进入项目目录；
3. 确认 Git 状态；
4. 运行 `codex`；
5. 输入任务；
6. 查看 Codex 的执行过程和结果；
7. 检查 diff；
8. 运行测试；
9. 决定是否保留改动。

示例：

```bash
cd my-project
git status
codex
```

然后输入：

```text
请阅读这个项目，说明主要模块、启动命令和测试命令。不要修改文件。
```

#### CLI 适合的任务

CLI 特别适合和命令行强相关的任务，例如：

```text
请运行测试，找出失败原因，并做最小修复。
```

```text
请检查 package.json scripts，告诉我这个项目有哪些常用开发命令。
```

```text
请运行 typecheck，修复类型错误，不要改变业务逻辑。
```

```text
请分析 CI 失败日志，并给出修复方案。
```

#### CLI 的优势

Codex CLI 的优势包括：

- 更适合开发者已有终端习惯；
- 更容易结合 shell、Git、测试命令使用；
- 适合远程环境和容器环境；
- 适合自动化和脚本化；
- 更接近真实工程执行环境。

#### CLI 使用注意事项

使用 CLI 时同样要注意安全边界：

- 确认当前目录是否正确；
- 确认 Git 工作区是否干净；
- 谨慎允许高风险命令；
- 对删除文件、安装依赖、修改配置等操作保持审查；
- 不要在含有敏感数据的目录中随意运行；
- 对生产环境操作保持人工确认。

### 2.4 Codex Cloud：浏览器中的云端开发代理

Codex Cloud 是云端使用方式，适合让 Codex 在浏览器中处理 GitHub 仓库任务。它可以在后台运行任务，查看日志，生成 diff，并协助创建 PR。

#### 适合人群

Codex Cloud 适合：

- 希望让 Codex 异步处理任务的开发者；
- 使用 GitHub 管理代码的团队；
- 想让 Codex 在云端创建 PR 的用户；
- 需要在多个任务之间切换的人；
- 希望把 Codex 接入团队协作流程的人。

#### 基本使用流程

Codex Cloud 的基本流程是：

1. 打开 `chatgpt.com/codex`；
2. 登录账号；
3. 连接 GitHub 仓库；
4. 设置任务运行环境；
5. 输入任务；
6. 查看执行日志；
7. 审查 Codex 生成的 diff；
8. 创建 PR 或继续修改。

#### 适合 Codex Cloud 的任务

Codex Cloud 很适合异步、仓库级任务，例如：

```text
请阅读这个仓库，解释整体架构，并指出最重要的模块。
```

```text
请修复 issue #123 中描述的问题，做最小改动，并补充测试。
```

```text
请根据这个需求实现一个小功能，并创建 PR。
```

```text
请分析当前应用架构中最可能导致故障的几个点，并给出改进建议。先不要修改代码。
```

#### 通过 GitHub PR 使用 Codex

官方还支持在 GitHub PR 评论中通过 `@codex` 委托任务。例如：

```text
@codex review this PR for correctness, security issues, and missing tests.
```

或者：

```text
@codex please fix the failing tests in this PR.
```

这让 Codex 可以更自然地进入团队协作流程。

#### Codex Cloud 的优势

Codex Cloud 的优势包括：

- 不依赖本地机器持续运行；
- 适合后台处理任务；
- 更容易与 GitHub PR 流程结合；
- 可以查看任务日志和 diff；
- 适合团队异步协作。

#### Codex Cloud 的注意事项

使用云端任务时，需要特别关注：

- 仓库权限；
- GitHub 授权范围；
- 云端运行环境是否与本地一致；
- 是否允许访问互联网；
- 是否涉及敏感文件或密钥；
- Codex 生成的 PR 是否经过人工 review。

### 2.5 四种入口如何选择

不同入口没有绝对优劣，关键是看你的工作场景。

| 使用场景 | 推荐入口 | 原因 |
|---|---|---|
| 第一次体验 Codex | Codex App | 图形界面直观，适合快速上手 |
| 日常编码 | IDE Extension | 可以和编辑器工作流结合 |
| 终端重度使用 | Codex CLI | 适合命令行、测试、脚本化工作流 |
| 异步处理 GitHub 仓库任务 | Codex Cloud | 适合后台运行、生成 diff 和 PR |
| PR 代码审查 | GitHub 集成 / Cloud | 更贴近团队协作流程 |
| CI 失败分析 | GitHub Action / CLI | 适合结合日志和测试命令 |
| 企业内部平台集成 | SDK / Agents SDK | 适合二次开发和平台化 |

如果你不知道该选哪个，可以按下面的顺序开始：

1. **先用 Codex App 理解基本体验**；
2. **再用 IDE Extension 融入日常编码**；
3. **如果你喜欢终端，再学习 Codex CLI**；
4. **如果你需要云端异步任务，再使用 Codex Cloud**；
5. **如果你是团队负责人，再研究 GitHub、Slack、CI/CD 和 SDK 集成**。

### 2.6 第一次使用 Codex 的推荐路径

对于完全从 0 开始的用户，推荐按下面路径实践。

#### 第一步：选择一个低风险项目

不要一开始就在核心生产项目中测试 Codex。你可以选择：

- 一个 demo 项目；
- 一个个人项目；
- 一个小型内部工具；
- 一个已经有 Git 管理的项目。

#### 第二步：先做只读任务

第一次不要让 Codex 修改文件。先让它理解项目。

```text
请先不要修改文件。请阅读这个项目，告诉我项目用途、技术栈、目录结构、启动方式、测试方式和关键文件。
```

#### 第三步：做一个低风险修改

例如修改 README、补充注释、修复拼写错误、增加一个简单测试。

```text
请根据 package.json 中的 scripts，在 README 中补充本地启动和测试说明。不要新增不存在的命令。
```

#### 第四步：查看 diff

检查 Codex 修改了哪些文件，是否符合预期。

```bash
git diff
```

#### 第五步：运行测试

如果项目有测试，运行相关命令。

```bash
npm test
```

或者：

```bash
npm run typecheck
npm run lint
```

#### 第六步：保留或回滚

如果结果满意，可以提交：

```bash
git add .
git commit -m "docs: update development instructions"
```

如果不满意，可以回滚：

```bash
git restore .
```

### 2.7 本章小结

本章介绍了 Codex 的四种主要使用入口：

- **Codex App**：适合新手和图形界面用户；
- **IDE Extension**：适合日常编码和局部开发任务；
- **Codex CLI**：适合终端、测试、脚本化和自动化工作流；
- **Codex Cloud**：适合 GitHub 仓库、异步任务、PR 和团队协作。

对于初学者，最稳妥的路径是：

```text
只读理解项目 → 小范围修改 → 查看 diff → 运行测试 → 人工 review → 提交或回滚
```

只要你从一开始就建立这个流程，Codex 就不会只是一个“会写代码的聊天工具”，而会逐渐成为你开发流程中的可靠协作者。
