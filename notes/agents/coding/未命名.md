# 第一章：认识 Codex —— 核心概念与架构

> **本章目标**：建立对 Codex 的正确心智模型，搞清楚它是什么、能做什么、三种形态如何配合，以及它的基本工作原理。读完本章，你将对 Codex 的整体架构有清晰的认知，不再把它当作"加强版 ChatGPT"来使用。

---

## 1.1 Codex 是什么：编程 Agent 与传统 AI 的本质区别

很多人第一次接触 Codex，会下意识地把它当成"更聪明的代码补全工具"，或者"会写代码的聊天机器人"。这两种理解都不准确，而且会直接导致你用不好它。

**Codex 的准确定义**：它是 OpenAI 推出的**编程 Agent**，能够**读取、修改和运行代码**。

这里有三个关键动词：**读取（read）、修改（modify）、运行（run）**。

### 传统 AI 助手 vs. Codex Agent

| 维度 | 传统 AI 助手（如 ChatGPT）| Codex Agent |
|---|---|---|
| 交互方式 | 对话，你问它答 | 任务委托，它自主执行 |
| 能否访问文件 | 不能（除非你手动粘贴） | 能，直接读取你的代码库 |
| 能否执行命令 | 不能 | 能，在沙箱中运行 shell 命令 |
| 能否修改文件 | 不能 | 能，直接写入代码文件 |
| 工作时长 | 一问一答，秒级响应 | 可以持续工作数分钟甚至更长 |
| 上下文来源 | 你粘贴过来的文字 | 你的本地仓库、文件树、运行结果 |
| 验证能力 | 无法自我验证 | 可以运行测试、lint、编译来验证自己的输出 |

这个区别决定了你应该如何与 Codex 交互：**不是"给我写一个函数"，而是"帮我实现这个功能，写完跑一下测试"**。

### Codex 能做什么

- **理解陌生代码库**：把 Codex 扔进一个你刚接手的项目，让它先读代码再解释架构
- **实现新功能**：给出需求，Codex 自主定位相关文件、编写代码、运行验证
- **修复 Bug**：粘贴报错日志，Codex 定位问题、修改代码、验证修复
- **代码审查**：让 Codex 扮演 reviewer，扫描 diff 提出改进意见
- **重构代码**：大规模重命名、抽象、拆分模块等重复性工作
- **撰写文档**：自动为函数、模块生成注释和 README
- **处理 DevOps 工作**：配置文件修改、CI 脚本调整、依赖更新

### Codex 不擅长什么

- **需要实时人类决策的设计讨论**：架构选型、产品方向，这些还是你来
- **跨越清晰边界的超大型任务**：一次性"帮我重写整个系统"通常效果不好
- **没有测试/验证手段的任务**：Codex 需要能运行某种验证才能知道自己做对了

## 1.2 Codex 的四种形态详解

Codex 不是单一产品，而是**一套跨环境的编程 Agent 体系**，目前有四种形态，各有适用场景。

```
┌─────────────────────────────────────────────────────────────┐
│                     OpenAI Codex                            │
│                                                             │
│   ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│   │  CLI     │  │  IDE     │  │  Cloud   │  │   App    │  │
│   │ 命令行   │  │ 编辑器   │  │  云端    │  │  桌面版  │  │
│   └──────────┘  └──────────┘  └──────────┘  └──────────┘  │
│                                                             │
│        四种形态共享同一套配置体系（config.toml）             │
└─────────────────────────────────────────────────────────────┘
```

### 形态一：Codex CLI（命令行工具）

**核心定位**：在你的本地终端里运行，直接作用于你机器上的代码。

- **运行环境**：本地，macOS / Linux（Windows 建议 WSL）
- **开源**：是，使用 Rust 编写，托管于 [github.com/openai/codex](https://github.com/openai/codex)
- **交互方式**：全屏 TUI（终端用户界面）或单次执行（one-shot）
- **典型场景**：日常编码、调试、重构、脚本化自动化
- **核心优势**：低延迟、本地文件直接访问、可脚本化、最灵活

```bash
# 启动交互式 TUI
codex

# 一次性执行不进入 TUI
codex "解释一下这个项目的目录结构"

# 脚本化执行
codex exec --approval-mode full-auto -- "跑所有单元测试并修复失败的用例"
```

### 形态二：IDE Extension（编辑器扩展）

**核心定位**：嵌入你的编辑器，在写代码时随时唤起 Codex。

- **支持编辑器**：VS Code、Cursor、Windsurf、VS Code Insiders
- **默认模式**：Agent 模式（可读文件、做编辑、运行命令）
- **与 CLI 关系**：共享同一套 `config.toml` 配置
- **典型场景**：开发过程中的即时协助、局部修改、上下文感知补全
- **核心优势**：不用离开编辑器、可直接 undo 编辑、与 Cloud 任务联动

### 形态三：Codex Cloud（云端 Agent）

**核心定位**：在 OpenAI 托管的隔离容器中运行，完全异步，主要与 GitHub 集成。

- **访问入口**：[chatgpt.com/codex](https://chatgpt.com/codex)
- **运行环境**：OpenAI 云端隔离容器（非本地）
- **与 GitHub 深度集成**：绑定仓库、自动创建 PR、支持 `@codex` 触发
- **典型场景**：独立新功能开发、后台长时任务、CI 失败自动修复
- **核心优势**：不占用本地资源、可并行多任务、天然隔离不影响本地开发

```
# 在 GitHub PR 评论中直接触发
@codex 这个测试失败了，帮我找出原因并修复
```

### 形态四：Codex App（桌面应用）

**核心定位**：专为多任务并行设计的桌面体验，附带 Computer Use 能力。

- **平台**：macOS / Windows
- **独特能力**：可以控制 macOS 原生应用和浏览器（Computer Use）
- **内置**：Git Worktree、内置浏览器、自动化（定时任务）、图片生成
- **典型场景**：同时跑多个独立任务、需要 GUI 交互的自动化
- **核心优势**：并行多线程任务、视觉交互能力

### 四种形态的配合关系

在实际工作中，这四种形态不是互斥的，而是**协同**的。一个典型的工作流：

```
1. 用 CLI 在本地快速探索和原型开发
         ↓
2. 用 IDE Extension 辅助迭代细节
         ↓
3. 把独立子任务（加认证、加数据库）委托给 Cloud Agent 并行处理
         ↓
4. Cloud Agent 创建 PR → 在 GitHub 审查 diff → 合并
         ↓
5. 在 IDE 继续下一个任务，同时 Cloud 跑另一个任务
```

## 1.3 账号与权限：哪个套餐能用什么

Codex 的访问权限与你的 OpenAI 账号套餐直接挂钩。

### 套餐与访问权限对照

| 套餐                     | CLI      | IDE Extension | Cloud Agent | App |
| ---------------------- | -------- | ------------- | ----------- | --- |
| **ChatGPT Plus**       | ✅        | ✅             | ✅           | ✅   |
| **ChatGPT Pro**        | ✅        | ✅             | ✅           | ✅   |
| **ChatGPT Business**   | ✅        | ✅             | ✅           | ✅   |
| **ChatGPT Edu**        | ✅        | ✅             | ✅           | ✅   |
| **ChatGPT Enterprise** | ✅        | ✅             | ✅           | ✅   |
| **OpenAI API Key**     | ✅（需额外配置） | ✅（需额外配置）      | ❌           | ❌   |
| **免费账号**               | ❌        | ❌             | ❌           | ❌   |

### 两种认证方式的区别

**方式一：ChatGPT 账号登录（推荐）**
- 最快捷，使用已购买计划内的额度
- 启动 CLI 后直接 OAuth 登录，无需管理 API Key
- 适合大多数个人用户和团队

**方式二：OpenAI API Key**
- 需要独立配置，按 token 计费
- 适合需要精细控制成本、集成到 CI 系统的场景
- 部分高级功能（Cloud Agent）不支持

> **实用建议**：刚开始使用时，直接用 ChatGPT 账号登录最省心。如果你需要把 Codex 嵌入 CI/CD 流水线，再考虑切换到 API Key 模式。

## 1.4 Codex 如何工作：核心机制解析

### 工作流程概览

```
你的 Prompt
     │
     ▼
┌─────────────────────────────────────────────────────┐
│                  Codex Agent                        │
│                                                     │
│  1. 读取上下文（代码库、AGENTS.md、配置、记忆）      │
│  2. 制定执行计划                                     │
│  3. 循环执行：                                       │
│     ├── 调用工具（读文件、写文件、运行命令、搜索）    │
│     ├── 检查结果                                     │
│     └── 决定下一步（或请求审批）                     │
│  4. 输出最终结果                                     │
└─────────────────────────────────────────────────────┘
     │
     ▼
代码变更 / 运行结果 / PR
```

### 三大核心机制

**① 工具调用（Tool Use）**

Codex 通过调用一系列内置工具来完成任务：

| 工具类型 | 作用 |
|---|---|
| 文件读取 | 扫描目录、读取源代码 |
| 文件写入 | 创建、修改、删除文件 |
| Shell 执行 | 运行命令、执行测试、调用编译器 |
| Web 搜索 | 查询最新文档、库版本信息 |
| MCP 工具 | 连接第三方系统（数据库、Notion、Linear 等）|
| 子 Agent | 派生专项 Agent 并行处理子任务 |

**② 沙箱机制（Sandbox）**

Codex 运行命令时不是直接在你的系统上运行，而是在**受控沙箱**中执行：

- **macOS**：使用 Apple Seatbelt（`sandbox-exec`）限制文件系统和网络访问
- **Linux**：使用 Landlock + seccomp 内核机制隔离
- **Cloud**：OpenAI 托管的完全隔离容器

沙箱的核心作用：**防止 Codex 意外（或被恶意 prompt 诱导）做出超出预期的操作**。

**③ 审批机制（Approvals）**

当 Codex 想执行某个操作，审批模式决定它是否需要先问你：

```
Codex 想执行某操作
         │
         ▼
    ┌────────────────────────────────────┐
    │ 审批模式判断                        │
    │                                    │
    │  Auto（默认）                       │
    │  ├── 沙箱内操作 → 自动执行          │
    │  └── 越出沙箱  → 请求你审批         │
    │                                    │
    │  Read-only → 只读，永不审批         │
    │                                    │
    │  Full Access → 全自动，不审批       │
    └────────────────────────────────────┘
```

> **关键理解**：沙箱决定"能做什么范围"，审批决定"超出范围时怎么处理"。两者配合使用。

### Codex 如何理解你的项目

Codex 在启动时会按顺序读取多层上下文：

```
启动顺序（从低优先级到高优先级）
──────────────────────────────
1. 全局 AGENTS.md（~/.codex/AGENTS.md）
          ↓
2. 仓库根目录 AGENTS.md
          ↓
3. 当前子目录 AGENTS.md
          ↓
4. 用户配置（~/.codex/config.toml）
          ↓
5. 项目配置（.codex/config.toml）
          ↓
6. CLI 参数（最高优先级）
```

`AGENTS.md` 是让 Codex"记住你的规则"的关键文件，例如：测试命令是什么、PR 应该怎么写、哪些命令禁止使用。我们在第七章会深入讲解。
