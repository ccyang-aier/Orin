## 1. vLLM 项目概述

### 1.1 什么是 vLLM？

vLLM 是一个**高性能、高吞吐量**的大语言模型（LLM）推理和服务引擎。其核心创新点包括：

| 核心特性               | 描述                                      |
| ------------------ | --------------------------------------- |
| **PagedAttention** | 借鉴操作系统虚拟内存管理思想，高效管理 KV Cache 内存         |
| **连续批处理**          | 动态调度请求，最大化 GPU 利用率                      |
| **分布式推理**          | 支持张量并行(TP)、流水线并行(PP)、数据并行(DP)、上下文并行(CP) |
| **多模态支持**          | 支持视觉-语言模型、音频模型等多种模态                     |

### 1.2 项目定位

## 3. 多进程架构详解

### 3.1 为什么采用多进程架构？

vLLM V1 采用**多进程架构**分离关注点，主要原因包括：

1. **GIL 规避**：Python 的全局解释器锁（GIL）限制了多线程的并行性能，多进程可以绑过这一限制
2. **内存隔离**：每个 Worker 进程管理独立的 GPU 内存，避免内存竞争
3. **故障隔离**：单个 Worker 的崩溃不会直接导致整个系统崩溃
4. **分布式支持**：多进程天然支持多节点分布式部署

### 3.3 进程数量计算

对于部署配置：`N` 个 GPU，`TP` 张量并行大小，`PP` 流水线并行大小，`DP` 数据并行大小：

| 进程类型 | 数量 | 计算公式 | 说明 |
|---------|------|----------|------|
| API Server | `A` | 默认等于 `DP` | 处理 HTTP 请求和输入处理 |
| Engine Core | `DP` | `data_parallel_size` | 调度器和 KV Cache 管理 |
| GPU Worker | `N` | `DP × PP × TP` | 每个 GPU 一个，执行模型前向传播 |
| DP Coordinator | 0 或 1 | `DP > 1 ? 1 : 0` | 跨 DP ranks 负载均衡 |

**总进程数计算公式**：
```
总进程数 = API进程数 + DP + N + (DP > 1 ? 1 : 0)
```

## 4. 并行策略与部署模式

### 4.1 并行策略概览

vLLM 支持多种并行策略，可以组合使用以适应不同规模的模型和硬件配置：

| 并行策略 | 缩写 | 参数名 | 核心思想 | 适用场景 |
|---------|------|--------|---------|---------|
| **张量并行** | TP | `tensor_parallel_size` | 将模型层内参数切分到多 GPU | 单层参数量大的模型 |
| **流水线并行** | PP | `pipeline_parallel_size` | 将模型层间切分到多 GPU | 层数多的模型 |
| **数据并行** | DP | `data_parallel_size` | 多份模型副本处理不同请求 | 高吞吐量场景 |
| **专家并行** | EP | `enable_expert_parallel` | 将 MoE 专家分布到多 GPU | MoE 模型 |
| **预填充上下文并行** | PCP | `prefill_context_parallel_size` | 将长序列切分到多 GPU | 长上下文 prefill 阶段 |
| **解码上下文并行** | DCP | `decode_context_parallel_size` | 解码阶段序列并行 | 长上下文 decode 阶段 |

### 4.2 张量并行 (Tensor Parallelism, TP)

#### 核心原理

张量并行将**模型层内的参数**切分到多个 GPU 上，每个 GPU 只保存部分权重，计算时通过通信聚合结果。

```
                        张量并行示意图 (TP=4)
                        
┌─────────────────────────────────────────────────────────────────────┐
│                     单层线性层 (Y = XW)                              │
│                                                                     │
│   输入 X                                                            │
│     │                                                               │
│     ▼                                                               │
│   ┌───────────────────────────────────────────────────────────┐    │
│   │                    All-Gather                              │    │
│   └───────────────────────────────────────────────────────────┘    │
│     │                                                               │
│     ├────────────┬────────────┬────────────┬────────────┐          │
│     ▼            ▼            ▼            ▼            │          │
│  ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐          │          │
│  │GPU 0 │    │GPU 1 │    │GPU 2 │    │GPU 3 │          │          │
│  │ W₀   │    │ W₁   │    │ W₂   │    │ W₃   │          │          │
│  │ Y₀   │    │ Y₁   │    │ Y₂   │    │ Y₃   │          │          │
│  └──┬───┘    └──┬───┘    └──┬───┘    └──┬───┘          │          │
│     │            │            │            │            │          │
│     └────────────┴────────────┴────────────┘            │          │
│                         │                                 │          │
│                         ▼                                 │          │
│   ┌───────────────────────────────────────────────────────────┐    │
│   │                 All-Reduce (求和)                          │    │
│   └───────────────────────────────────────────────────────────┘    │
│                         │                                           │
│                         ▼                                           │
│                     输出 Y                                           │
```

#### TP 组的形成

在 vLLM 中，TP 组的形成规则如下（参考 `parallel_state.py`）：

```python
# 假设 world_size = 8, TP = 2, PP = 2, DP = 2
# all_ranks 的形状为 (DP, PP, PCP, TP) = (2, 2, 1, 2)
all_ranks = torch.arange(8).reshape(2, 2, 1, 2)

# TP 组通过 view + unbind 形成
# 结果: [[0, 1], [2, 3], [4, 5], [6, 7]]
group_ranks = all_ranks.view(-1, tensor_parallel_size).unbind(0)
```

### 4.3 流水线并行 (Pipeline Parallelism, PP)

#### 核心原理

流水线并行将**模型的不同层**分配到不同的 GPU 上，数据像流水线一样依次经过各阶段。

```
                    流水线并行示意图 (PP=4)
                    
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│  时间步 T1:                                                          │
│  ┌──────┐                                                           │
│  │GPU 0 │ ←──── 输入                                                │
│  │层0-7 │                                                           │
│  └──────┘                                                           │
│                                                                     │
│  时间步 T2:                                                          │
│  ┌──────┐    ┌──────┐                                               │
│  │GPU 0 │ → │GPU 1 │                                               │
│  │层0-7 │    │层8-15│                                               │
│  └──────┘    └──────┘                                               │
│     ↑                                                               │
│   新输入                                                            │
│                                                                     │
│  时间步 T3:                                                          │
│  ┌──────┐    ┌──────┐    ┌──────┐                                   │
│  │GPU 0 │ → │GPU 1 │ → │GPU 2 │                                   │
│  │层0-7 │    │层8-15│    │层16-23│                                  │
│  └──────┘    └──────┘    └──────┘                                   │
│     ↑                                                               │
│   新输入                                                            │
│                                                                     │

│  ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐                       │

│  │层0-7 │    │层8-15│    │层16-23│   │层24-31│                      │
│  └──────┘    └──────┘    └──────┘    └──────┘                       │
│     ↑                                                               │

│  稳态: 所有 GPU 同时工作，吞吐量最大化                               │
```

#### PP 组的形成

```python
# PP 组通过 transpose + reshape 形成
# 假设 all_ranks 形状为 (DP, PP, PCP, TP)
group_ranks = (
    all_ranks.transpose(2, 4)  # 将 PP 维度移到末尾
    .reshape(-1, pipeline_model_parallel_size)
    .unbind(0)
)
```

#### PP 的特殊角色

在 PP 中，有特殊角色区分：

| 角色 | 判断条件 | 职责 |
|------|---------|------|
| **First Rank** | `PP rank == 0` | 接收输入 embeddings，开始前向传播 |
| **Last Rank** | `PP rank == PP_size - 1` | 执行采样，产生最终输出 token |
| **Middle Rank** | 其他 | 接收上一阶段中间结果，传递给下一阶段 |

```python
# 在 gpu_worker.py 中的处理逻辑
if forward_pass and not get_pp_group().is_first_rank:
    # 非第一个 PP rank 需要接收上一阶段的数据
    tensor_dict = get_pp_group().irecv_tensor_dict(...)
    
if not get_pp_group().is_last_rank:
    # 非最后一个 PP rank 需要发送数据给下一阶段
    get_pp_group().isend_tensor_dict(output.tensors, ...)
```

### 4.4 数据并行 (Data Parallelism, DP)

#### 核心原理

数据并行在多个 GPU 上**复制完整的模型**，每个 GPU 处理不同的请求批次，实现请求级别的并行。

```

│   │  DP=0    │ │  DP=1    │  │  DP=2    │ │  DP=3    │            │
│   │          │ │          │  │          │ │          │            │
│   │ 完整模型 │ │ 完整模型 │  │ 完整模型 │ │ 完整模型 │            │
│   │ 副本 A   │ │ 副本 B   │  │ 副本 C   │ │ 副本 D   │            │

│   └──────────┘ └──────────┘  └──────────┘ └──────────┘            │
│                                                                     │
│   特点: 每个 DP rank 独立处理请求，吞吐量线性增长                    │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

#### DP 的特殊性

与其他并行方式不同，DP 不需要在模型计算过程中进行通信。但是：

1. **MoE 模型**：需要跨 DP ranks 进行专家同步（当使用 EP 时）
2. **负载均衡**：需要协调各 DP rank 的请求分配
3. **内存同步**：某些场景需要同步 KV Cache 内存大小

```python
# parallel_state.py 中的 DP 同步方法
@staticmethod
def sync_kv_cache_memory_size(dp_group: ProcessGroup, kv_cache_memory: int) -> int:
    # 跨 DP ranks 同步 KV Cache 内存大小，取最小值
    torch.distributed.all_reduce(tensor, op=ReduceOp.MIN, group=dp_group)
    return tensor.item()
```

### 4.6 上下文并行 (Context Parallelism, CP)


## 5. 核心组件详解

### 5.1 组件概览

| 组件 | 所在文件 | 核心职责 | 存在数量 |
|------|---------|---------|---------|
| **Executor** | `v1/executor/` | 管理 Worker 生命周期，协调分布式执行 | 1 个（per Engine Core） |

### 5.5 Scheduler (调度器)

#### 职责

Scheduler 是 vLLM 的核心调度组件：

1. **请求队列管理**：维护 waiting、running、swapped 队列
2. **内存分配**：为请求分配 KV Cache 块
3. **调度策略**：决定哪些请求可以被调度
4. **抢占与恢复**：在内存不足时抢占低优先级请求
5. **前缀缓存**：检测和利用前缀缓存命中

#### 调度循环

```python
# scheduler.py (简化版)
class Scheduler:
    def schedule(self) -> SchedulerOutput:
        # 1. 处理已完成的请求
        self._process_finished_requests()
        
        # 2. 尝试调度 waiting 队列中的请求
        while self.waiting:
            request = self.waiting[0]
            
            # 检查是否可以分配内存
            if self._can_allocate(request):
                self._allocate_and_schedule(request)
                self.waiting.popleft()
                self.running.append(request)
            else:
                # 内存不足，尝试抢占
                self._preempt_if_needed()
                break
        
        # 3. 返回调度结果
        return SchedulerOutput(
            scheduled_requests=...,
            num_scheduled_tokens=...,
        )
```

### 5.6 KVCacheManager (KV Cache 管理器)

### 6.4 Driver Worker 概念

在 vLLM 中，有一个特殊的 **Driver Worker** 角色：

1. **响应收集**：只有 Driver Worker 返回模型输出结果

```
│   │              PP=0              │              PP=1           │   │
│   │        ┌─────────┬─────────┐   │        ┌─────────┬───────┐ │   │
│   │        │   TP=0  │   TP=1  │   │        │  TP=0  │  TP=1  │ │   │
│   │        │ Rank=0  │ Rank=1  │   │        │ Rank=2 │ Rank=3 │ │   │
│   │        │ ★Driver │         │   │        │ ★Driver│        │ │   │
│   │        └─────────┴─────────┘   │        └─────────┴───────┘ │   │
│   └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│   ★ Driver Worker = TP 组内 rank % TP_size == 0 的 Worker          │
│                                                                     │
│   在 PP 的最后一个 stage (PP=1)，只有 Rank=2 的 Driver Worker       │
│   会执行采样并返回最终输出。                                         │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 6.5 Output Rank 确定

在分布式推理中，需要确定哪个 Rank 返回结果：

```python
# multiproc_executor.py
def _get_output_rank(self) -> int:
    # 返回 TP rank=0 且 PP rank=-1 的 Worker
    # 即最后一个 PP stage 的第一个 TP Worker
    return (
        self.world_size
        - self.parallel_config.tensor_parallel_size
        * self.parallel_config.prefill_context_parallel_size
    )
```

## 总结

本文档系统介绍了 vLLM 的核心架构概念：

1. **多进程架构**：vLLM 采用 API Server、Engine Core、Worker 多进程分离的架构，实现关注点分离和高性能推理。

2. **并行策略**：vLLM 支持 TP、PP、DP、EP、CP 等多种并行策略，可以灵活组合以适应不同规模模型和硬件配置。

3. **核心组件**：
   - Executor：分布式执行协调者
   - Worker：GPU 设备管理和模型执行载体
   - ModelRunner：模型执行逻辑封装
   - Scheduler：请求调度和内存管理
   - KVCacheManager：KV Cache 生命周期管理

4. **Rank 体系**：理解 Global Rank、Local Rank、Rank in Group 的区别是理解分布式通信的基础。

理解这些概念后，您可以继续深入学习 vLLM 的其他主题，如 PagedAttention 实现、CUDA Graphs 优化、量化支持等。
