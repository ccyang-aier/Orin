# FOAI · 易用，竞争力落地及持续演进进展

## P1 重新定义FOAI易用性

在AI Infra时代，易用性已从降低系统操作门槛转向系统是否可理解、可控、可托付、可进化。

| 过去的关注点      | 正在转向                             |
| ----------- | -------------------------------- |
| 页面操作简单、引导清晰 | 系统行为可理解、可解释                      |
| 功能全、模块多     | 意图可表达、决策有依据                      |
| 看指标、看日志、看告警 | 给原因、给建议、给动作                      |
| 固定WebUI     | GUI · CLI · 自然语言 · Agent调用等统一操作面 |
| 经验积累在人身上    | 经验沉淀在平台，可复用、可进化                  |

## P2 FOAI易用性三层演进


L1

可操作（Operable）

流程可走通，核心功能覆盖完整，但依赖用户主动查文档、试错调参、基于日志、监控等分析定位问题。

状态：已基本完成，存在局部短板


L2

可理解（Understandable）

系统主动透出状态、归因根因、引导决策，用户不需要猜，平台主动说话帮助用户决策，复杂性被吸收。

状态：26已规划部分需求正在该方向努力，如智能端到端推理拓扑。


L3

意图驱动（Intent-Driven）

用户只需描述目标，平台自动推导配置、自愈异常、故障闭环，沉淀经验，用户业务可托付，平台智能可进化。

状态：/

当前，FOAI正处于L1到L2的过渡期。


## P3 FOAI历史使用问题总览

基于AI4RD用户真实反馈：

| 问题类别                 | 数量  | 占比      | 代表现象                |
| -------------------- | --- | ------- | ------------------- |
| 性能问题（推理慢 / 超时 / 死循环） | 47  | **46%** | 任务堆积、上下文超限、卡顿无响应    |
| 变更故障（版本 / 环境变更引发）    | 17  | **17%** | 框架升级致接口不兼容、服务中断     |
| 需求（功能缺失）             | 14  | **14%** | 错误提示不清、统计维度不足、接口不支持 |
| 服务中断                 | 10  | **10%** | 节点下线、环境更新、硬件故障      |
| 配置问题                 | 7   | **7%**  | 启动参数未启用、上下文设置不当     |
| 操作失误                 | 2   | **2%**  | 磁盘空间不足、模型权重校验失败     |

在上百条反馈中，70%+的问题不是功能缺失，而是用户不理解系统行为（性能问题46% + 配置问题7% + 变更故障17%），本质即用户对系统存在较严重的认知负担：

| 认知负担               | 含义                       | 在反馈中的比重               |
| ------------------ | ------------------------ | --------------------- |
| 不可见（Invisible）     | 知道"慢/错/超时"，但不知道为什么，根因不透出 | 性能46% + 服务中断10% ≈ 56% |
| 不可决策（Undecidable）  | 模型体验、评估、部署数据分散，最终靠经验拍板   | 配置7% + 变更17% ≈ 24%    |
| 不可闭环（Unactionable） | 看到了异常，但无法顺着问题直接进入修复动作    | 贯穿所有类别                |

结论：FOAI当前处于可用但不足够好用阶段，针对完备可观测、可解释、决策辅助、可进化等智能运维能力维度上均存在较大差距。
