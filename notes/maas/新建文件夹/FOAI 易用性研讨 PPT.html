01
破题
易用，不是体验修补
易用性的竞争，已经不是 UI 好不好看，而是平台能不能把 AI 的复杂性吸收掉，把正确结果更稳定地交给用户。
用户真实处境
  • 每一步都不确定"这样配对不对"
  • 出了问题不知道为什么、不知道该怎么办
  • 评估了一堆模型，最终还是靠猜
  • 每个模块能用，但连起来走一遍很费劲
核心命题

02
趋势判断
易用性的竞争主战场，已经迁移
从"降低页面操作门槛"，升级到"吸收 AI Infra / Agent Infra 的系统复杂性"。
过去的竞争重点 → 正在成为新重点
页面操作简单、引导清晰 → 系统行为可理解、可解释
功能全、模块多 → 意图可表达、决策有依据
看指标、看日志 → 给原因、给建议、给动作
固定 WebUI → GUI · 自然语言 · Agent 调用统一操作面

行业支撑：Agent 时代难点从"生成答案"转向"约束行为"；Memory 升级为系统能力；Observability 从看指标升级为解释系统行为；用户的不安，本质是心智模型不匹配。

04
现状诊断
不是功能不够，是存在四个断裂
FOAI 当前的易用性问题，本质是四个系统性断裂，而不是功能缺失。
认知断裂
用户看到慢、看到错，但不知道为什么。配置参数没有语义，排障路径靠经验积累。
决策断裂
评估、体验、部署、运行是四个孤岛。每次部署决策，用户还是在靠猜。
处置断裂
看到异常到行动之间，有大量手动跳转。告警触发了，但没有处置路径。
进化断裂
成功经验、踩坑案例、最佳配置无法沉淀。每个新用户都要从零开始摸索。

数据来源：AI4RD 用户反馈高频词 — 错误信息不可读 / 任务堆积无感知 / 配置黑盒 / 告警混乱 / 跨模块割裂

05
演进方向 01
从"部署工具"升级为"意图编译器"
部署易用的下一阶段，不是模板更多，而是让用户表达目标，平台负责编译成方案。
当前模式
手配参数
用户手配几十个底层参数：并发数、副本数、资源规格、超时时间……
配错了才知道，失败才报错。
→
目标模式
意图编译
用户表达业务意图 →
平台推理出推荐方案 / 风险评分 / 多方案对比 →
历史成功配置沉淀为推荐能力。
场景意图式部署 部署前风险评分 多方案对比推荐 历史经验 → 推荐起点
06
演进方向 02
从"监控平台"升级为"诊断副驾"
观测的终局不是看板更多，而是系统直接给原因、建议和动作。
当前模式
看指标·猜原因
异常发生 → 看告警 → 看指标 → 看日志 → 猜原因 → 尝试处理

"慢"只是现象，根因可能有 5 种以上。
→
目标模式
给原因·给动作
异常发生 → 系统语义化分析 → 输出因果诊断链
→ 给出处置建议 → 一键进入处置闭环
"慢"的语义拆解 故障因果图 告警语义分级 平台级故障语义层 告警 → 处置 一键闭环
