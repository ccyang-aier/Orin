# FOAI MaaS推理平台 — 易用性竞争力落地及持续演进

---

## 📌 P1 | 开篇：易用性的真问题

**主题定位**

"3+1竞争力"体系中，永快和永新解决的是**技术供给侧**的问题；
易用性解决的是**用户认知侧**的问题——它决定了技术能力能否被高效消费。

**一个核心观点**

> 易用性的本质不是"少点几下鼠标"，而是**认知负荷管理**——
> 降低用户在"理解系统→做出决策→完成操作"全链路中的认知成本。

**本次研讨目标**

- 不做特性清单汇报
- 聚焦三个问题：我们在哪？行业往哪走？我们怎么走？

## 📌 P2 | 业界洞察：三个正在发生的范式转变

### 转变一：UI不再是差异化因素

> "Surface-level design won't be enough to stay competitive."
> — NN/g, *State of UX in 2026*

- 设计系统标准化 + AI生成能力 → 任何团队都能产出"还不错"的界面
- 真正的差异化正在转向**系统行为层**：默认值是否智能、流程是否感知上下文、异常处理是否主动

**对FOAI的启示**：继续打磨UI控件的边际收益递减，需要向"系统智能行为"转移投入重心。

### 转变二：AI最成功的落地是认知增强，不是自主操作

> Thoughtworks 2025年 AIOps实践报告（16个客户、20个PoC、11个投产）：
> - 最高价值用例：**重复工单检测、运维知识检索、根因分析辅助**
> - L1/L2工单量降低 **35–40%**，RCA周期从小时级缩短到分钟级
> - 自主修复（auto-remediation）仍受限于风险治理，未规模化

**对FOAI的启示**：优先做"帮用户理解和决策"的AI能力，而非急于做"替用户执行"的Agent。

### 转变三：从AI-Bolted到AI-Native是架构级跃迁

> Taskade提出的**"AI移除测试"**：去掉AI功能后产品是否仍然完整？
> - Notion去掉AI → 仍是完整的文档工具（AI-Bolted）
> - Gartner预测：**40%的Agentic AI项目将因在遗留架构上修补而失败**（2027）

**对FOAI的启示**：我们需要判断——FOAI的AI能力是"锦上添花的附加层"，还是"平台不可分割的一部分"？

## 📌 P3 | 现状评估：FOAI易用性成熟度审视

### 已建立的基础（L1 — 操作效率层）

| 能力 | 状态 | 解决的问题 |
|------|------|-----------|
| 一键部署推理服务 | ✅ 已上线 | 降低部署操作步数 |
| 模型卡片化展示 | ✅ 已上线 | 降低模型选择的信息过载 |
| 资源拓扑可视化 | ✅ 已上线 | 降低运维排障的理解成本 |
| 统一APIKey管理 | ✅ 已上线 | 降低多服务接入的管理复杂度 |
| 引导式模型评估流程 | ✅ 已上线 | 降低评估任务的上手门槛 |

### 坦率的差距

- **智能程度不足**：表单填写仍依赖用户经验判断（如GPU类型选择、并发参数配置）
- **上下文断裂**：监控告警 → 定位原因 → 调整配置，三步之间缺乏信息连续性
- **知识沉淀缺失**：团队积累的最佳实践未结构化呈现在操作流程中

**一句话总结**：我们做好了"让操作变简单"，还没做到"让决策变容易"。

## 📌 P4 | 核心策略：认知负荷驱动的三层递进模型

```
┌─────────────────────────────────────────────────────┐
│  L3 · 架构跃迁层                                      │
│  双轨交互 · MCP统一接口 · Context Engineering          │
│  ─ 用户用自然语言表达意图，系统编排执行                    │
├─────────────────────────────────────────────────────┤
│  L2 · 决策辅助层                          ← 当前主攻  │
│  Smart Default · 上下文感知 · 知识主动呈现              │
│  ─ 系统替用户做80%的常规判断                            │
├─────────────────────────────────────────────────────┤
│  L1 · 操作效率层                          ← 已建立    │
│  流程精简 · 可视化 · 引导式交互                         │
│  ─ 用户知道该怎么做，系统让操作更快                      │
└─────────────────────────────────────────────────────┘
```

**每一层解决不同类型的认知负荷：**

| 层级 | 认知负荷类型 | 用户心理 | 解决思路 |
|------|-------------|---------|---------|
| L1 | 操作负荷 | "步骤太多，记不住" | 减少步骤，提供引导 |
| L2 | 决策负荷 | "选项太多，不确定该选哪个" | 智能推荐，替用户判断 |
| L3 | 表达负荷 | "我知道要什么，但不知道怎么告诉系统" | 自然语言交互 |

## 📌 P5 | L2重点攻坚：让系统替用户做80%的常规判断

### 5.1 Smart Default — 不是默认值，是"此时此刻的最优建议"

**传统默认值**：写死的固定值，与用户上下文无关
**Smart Default**：基于模型特征 + 集群资源 + 历史部署模式，动态生成推荐配置

> 示例：用户部署Qwen-72B
> - 系统识别模型参数量 → 推荐GPU类型和数量
> - 查询当前集群资源水位 → 推荐部署到资源充裕的节点组
> - 匹配历史同类部署 → 预填并发数、超时时间、量化策略
> - 用户只需确认或微调，而非从零填写

**技术依赖**：SmartModelHub模型元数据 + 集群状态API + 部署历史统计

### 5.2 上下文感知 — 打通操作链路的信息断裂

**当前痛点**：告警页面看到异常 → 切到监控页面查指标 → 切到服务详情改配置 → 回到告警确认

**目标体验**：
- 告警卡片内**直接关联**该服务的关键指标趋势
- 指标异常时**主动提示**可能原因（基于知识库匹配，非自由推理）
- 提供**一键跳转到精确操作位**，而非跳转到模块首页

### 5.3 知识主动呈现 — 在决策点嵌入团队经验

> Thoughtworks发现：AI最成功的运维用例是**知识检索**而非自主操作

- 将最佳实践结构化为**场景化知识片段**（如"72B模型部署注意事项"）
- 在相关操作节点**主动推送**，而非被动等用户搜索
- 来源标注清晰（"基于团队过去N次部署经验"），建立信任

**底层支撑**：LLM Wiki知识库 + Hermes语义检索能力（已在架构师输入方案中）

## 📌 P6 | L3架构跃迁：从"操作界面"到"意图界面"

### 6.1 双轨交互架构

```
用户
 ├── GUI轨道（结构化操作）
 │    表单、拓扑图、仪表盘 → 精确控制、批量操作、数据审阅
 │
 └── AI Native轨道（自然语言）
      对话界面 → 意图表达、跨模块查询、复杂编排
      
两条轨道共享同一数据层和权限体系，可随时切换
```

**为什么不是"全面转向对话式"？**

- NNGroup指出：视觉界面在**审阅、对比、批量操作**场景不可替代
- The Gradient指出：成功的产品将语音/自然语言作为**事务层**，视觉界面负责**验证层**
- 双轨不是过渡方案，而是**长期共存的终态架构**

### 6.2 MCP Server — 统一能力接口层

**价值定位**：将FOAI各模块能力（模型管理、服务部署、监控查询等）统一暴露为标准化接口

**务实判断**（引用Thoughtworks）：
> "MCP等Agent通信协议尚未工业化——上下文链不可控增长、编排复杂度高、执行路径不透明"

**我们的策略**：
- MCP Server作为**能力抽象层**先行建设，为AI交互提供稳定的工具集
- Agent编排层**渐进式引入**，初期限定在确定性高的场景（查询、推荐类）
- 避免在治理体系不成熟时让Agent执行高风险操作（如自动扩缩容）

### 6.3 自举优势 — MaaS平台独有的能力内化

> **关键洞察**：FOAI作为MaaS推理平台，运行着大量LLM推理服务。
> 这意味着我们可以将自身的推理能力内化到平台运维和体验中——
> **推理平台用自身的推理能力优化自身**，这是纯IaaS/PaaS平台不具备的结构性优势。

具体形式：
- 利用平台已部署的模型能力驱动Smart Default推理
- 利用平台推理服务运行知识检索和语义匹配
- 利用平台监控数据训练异常检测和容量预测模型

**这不是额外建设一套AI系统，而是让平台的核心资产为自身服务。**

## 📌 P7 | 竞争力协同：易用性如何驱动飞轮效应

```
            ┌──── 永快 ────┐
            │  高性能推理    │
            │  低延迟响应    │
            └──────┬───────┘
                   │  性能余量 → 支撑平台内AI推理
                   ▼
  ┌──── 易用 ──────────────────── 永新 ────┐
  │  Smart Default降低上手门槛      快速迭代模型支持  │
  │  知识呈现减少试错成本           新模型Day1可用    │
  │  自然语言降低表达门槛                             │
  └──────────────┬────────────────────────┘
                 │  更多用户 → 更多使用数据
                 ▼
           用户增长 & 粘性
                 │  使用数据 → Smart Default更准
                 │  用户反馈 → 知识库更丰富
                 └──→ 飞轮自增强
```

**核心逻辑**：
- **永快**提供性能余量，使平台内AI推理成为可能（自举优势的物质基础）
- **永新**保证新模型快速上架，易用性保证新模型能被快速消费
- **易用性**产生的使用数据反哺Smart Default和知识库，形成数据飞轮

**竞争壁垒**：这个飞轮是"用起来"之后才能转动的——易用性是飞轮的启动点。

## 📌 P8 | 路线图：确定性交付 + 探索性预研

### 近期确定性交付（Q2–Q3 2026）

| 事项 | 目标 | 度量标准 |
|------|------|---------|
| Smart Default v1 | 覆盖推理服务部署核心参数 | 用户手动修改率 < 30% |
| 告警-监控-操作上下文串联 | 告警到操作平均跳转次数降低 | 操作路径步数减少 ≥ 40% |
| 场景化知识片段嵌入 | Top 20高频操作场景覆盖 | 用户知识片段点击/展开率 |
| LLM Wiki知识库初始化 | 内部最佳实践结构化入库 | 覆盖率（场景数 / 总高频场景数）|

### 中期探索性预研（Q3–Q4 2026）

| 事项 | 目标 | 关键风险 |
|------|------|---------|
| MCP Server核心接口 | 5个核心模块能力接口化 | 接口粒度定义、权限模型 |
| 自然语言查询PoC | "我的服务A的P99延迟趋势" → 图表 | 意图识别准确率、响应延迟 |
| tCache → Agent记忆层升级 | 支撑多轮对话上下文保持 | 记忆淘汰策略、存储成本 |

### 度量体系

**北极星指标**：新用户首次成功部署推理服务的时间（Time-to-First-Deployment）

**辅助指标**：
- 操作效率：关键流程平均步数、表单手动填写字段数
- 决策质量：Smart Default采纳率、配置回滚率
- 知识消费：知识片段曝光→点击→采纳的转化漏斗
- 用户满意度：任务完成后即时满意度评分（CSAT）

## 📌 P9 | 总结：三个带走的观点

**观点一：易用性的战场正在从操作层转向决策层**

UI标准化和AI生成正在拉平操作层体验差距。
下一阶段的差异化在于：系统能否理解用户意图，主动减少决策负担。

**观点二：MaaS平台的自举优势是我们独有的结构性机会**

我们不需要额外采购或建设AI能力——平台自身运行的推理服务就是最好的资源。
让推理平台优化自身，这是竞对难以复制的壁垒。

**观点三：渐进务实，但方向明确**

- 近期集中在L2层（Smart Default + 上下文感知 + 知识呈现）——确定性高、见效快
- 中期布局L3层（双轨交互 + MCP）——架构先行，场景渐进
- 不做"全面Agent化"的冒进，引用Gartner数据：40%的Agentic AI项目因在遗留架构上修补而失败
