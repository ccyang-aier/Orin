## P1 重新定义FOAI易用性

在AI Infra时代，易用性已从降低系统操作门槛转向——**系统是否可理解、可控、可托付、可进化**。

| 过去的关注点      | 正在转向                             |
| ----------- | -------------------------------- |
| 页面操作简单、引导清晰 | 系统行为可理解、可解释                      |
| 功能全、模块多     | 意图可表达、决策有依据                      |
| 看指标、看日志、看告警 | 给原因、给建议、给动作                      |
| 固定 WebUI    | GUI · CLI · 自然语言 · Agent调用 统一操作面 |
| 经验积累在人身上    | 经验沉淀在平台，可复用、可进化                  |

这一转变表明：在AI Infra系统易用性的演进上，单纯传统思维xx正在向xx转变。

## P2 FOAI易用性三层演进模型

```
L1  可操作（Operable）
    流程可走通，核心功能覆盖完整。
    但依赖用户主动查文档、试错调参、基于日志和监控自行定位问题。
    状态：已基本完成，存在局部短板

        ↓

L2  可理解（Understandable）
    系统主动透出状态、归因根因、引导决策。
    用户不需要猜，平台主动"说话"，帮助用户理解系统在发生什么、该怎么做。
    复杂性被平台吸收，而非转嫁给用户。
    状态：2026年已规划部分需求，如智能端到端推理拓扑，正在该方向推进

        ↓

L3  意图驱动（Intent-Driven）
    用户只需描述目标，平台自动推导配置、自愈异常、故障闭环，沉淀经验。
    用户业务可托付，平台智能可进化。
    状态：探索中，与架构师提出的双轨交互、Harness Engineering方向高度对齐
```

**当前，FOAI正处于 L1 → L2 的过渡期，核心命题是：如何让平台"开口说话"。**

行业参照：
- **AWS Bedrock**：模型调用的配置错误会返回结构化 ActionableError，直接给出修复建议
- **Azure AI Foundry**：部署阶段提供参数约束可视化与冲突检测，拦截错误前置
- **Vertex AI**：推理监控与 SLO 自动健康度打分，异常自动关联根因分析

## P3 FOAI历史使用问题总览

基于 AI4RD 用户真实反馈（102条）：

| 问题类别 | 数量 | 占比 | 代表现象 |
|---|---|---|---|
| 性能问题（推理慢 / 超时 / 死循环） | 47 | **46%** | 任务堆积、上下文超限、卡顿无响应 |
| 变更故障（版本 / 环境变更引发） | 17 | **17%** | 框架升级致接口不兼容、服务中断 |
| 需求（功能缺失） | 14 | **14%** | 错误提示不清、统计维度不足、接口不支持 |
| 服务中断 | 10 | **10%** | 节点下线、环境更新、硬件故障 |
| 配置问题 | 7 | **7%** | 启动参数未启用、上下文设置不当 |
| 操作失误 | 2 | **2%** | 磁盘空间不足、模型权重校验失败 |

这些问题表面各异，用 L1/L2/L3 框架重新审视，根因收敛为**三类认知负担**：

| 认知负担                   | 含义                        | 在反馈中的比重               |
| ---------------------- | ------------------------- | --------------------- |
| **不可见**（Invisible）     | 知道"慢/错/超时"，但不知道为什么——根因不透出 | 性能46% + 服务中断10% ≈ 56% |
| **不可决策**（Undecidable）  | 模型体验、评估、部署数据分散，最终靠经验拍板    | 配置7% + 变更17% ≈ 24%    |
| **不可闭环**（Unactionable） | 看到了异常，但无法顺着问题直接进入修复动作     | 贯穿所有类别                |

如：
认知：知道“慢/错/超时”，但不知道为什么慢、为什么错、为什么变化
决策：模型体验、评估、部署、运行数据分散，最终仍靠经验拍板
行动：看到了异常，但无法顺着问题直接进入行动闭环
进化：成功部署经验、排障路径、最佳参数、常见坑没有沉淀为平台能力

**结论：FOAI当前最大的易用性缺口不是功能缺失，而是"系统不会说话"——现象有了，但原因没给、决策没帮、行动没引导、经验没沉淀。这正是 L2 的核心战场。**

## P3.5 四个演进方向

基于以上分析，FOAI 易用性演进围绕四个方向展开：

```
┌─────────────────────────────────────────────────────────┐
│                    FOAI 易用性演进四向                     │
│                                                         │
│  ① 智能诊断        ② 记忆沉淀        ③ 智能决策        ④ Agent易用  │
│  系统的眼睛        系统的经验        系统的大脑        系统的接口  │
│  （可观测性升级）   （知识资产化）    （AI Copilot）    （ACI设计） │
└─────────────────────────────────────────────────────────┘
```

四个方向相互依赖、递进：**智能诊断**是数据基础，**记忆沉淀**是知识积累，**智能决策**是能力输出，**Agent易用**是边界延伸。它们共同支撑 FOAI 从 L1 向 L3 演进。

## P4 FOAI · 智能诊断

### 当前观测体系

FOAI 现有观测能力已覆盖推理服务全链路，核心模块包括：

| 模块 | 当前能力 |
|---|---|
| 推理服务监控 | GPU利用率、QPS、P50/P99延迟、错误率实时展示 |
| 端到端拓扑 | 服务调用链路可视化，节点状态感知 |
| API日志 | 请求级别的入参/出参、耗时、状态码记录 |
| 告警体系 | 阈值触发告警，支持多渠道通知 |

现状：**能看到发生了什么，但无法回答为什么，更无法给出下一步该怎么做。**

---

### 演进方向：可观测性如何服务于易用性

在 AI Infra 与 Agent 时代，可观测性不再只是"监控数据的搜集与展示"，而是**平台智能的感知层**——它是系统 AI Copilot 的眼睛。没有高质量、结构化的可观测数据，后续的归因、决策、闭环都是无根之木。

**核心命题：如何构建一套面向智能的可观测框架？**

#### 框架设计思路

```
                    ┌─────────────────────────────┐
                    │     用户/Agent 感知层         │
                    │  "为什么慢？" "下一步怎么做？"  │
                    └──────────────┬──────────────┘
                                   │
                    ┌──────────────▼──────────────┐
                    │        归因与诊断层            │
                    │   结构化根因 · LLM归因引擎     │
                    │   AIOps：故障链路关联分析       │
                    └──────────────┬──────────────┘
                                   │
          ┌────────────────────────▼────────────────────────┐
          │                  数据语义化层                     │
          │  原始指标 → 结构化事件 → 可理解的上下文信号         │
          │  （时序数据、日志、Trace → 统一Schema）            │
          └────────────────────────┬────────────────────────┘
                                   │
          ┌────────────────────────▼────────────────────────┐
          │                  数据采集层（现有）               │
          │    监控指标 · Trace · 日志 · 拓扑 · 告警          │
          └─────────────────────────────────────────────────┘
```

**关键设计原则：**

1. **数据语义化（Semantic Observability）**：原始指标必须被翻译为可被 LLM 理解的结构化上下文。例如，"P99 延迟从 800ms 升至 3200ms，同期 GPU 显存占用率 97%，队列深度 142" 需封装为带语义的事件，而非散点指标。这是 Anthropic Context Engineering 原则在可观测性领域的直接应用——**Context is the new code**。

2. **Harness 架构接入**：可观测模块应作为 Harness 的标准数据源接入，以 MCP Server 形式向 AI Copilot 提供结构化的服务状态、异常事件、历史故障摘要，而不是让 LLM 直接读原始日志。

3. **故障链路 LLM 归因（AIOps）**：参考 ACM Computing Surveys 2025 《A Survey of AIOps in the Era of Large Language Models》的研究结论——LLM 在异常检测、根因分析、故障处置三个环节均已有工程落地案例。FOAI 可以从"故障归因"切入：当告警触发时，自动聚合拓扑快照 + 指标趋势 + 近期变更记录，生成结构化根因摘要，直接输出给用户。

4. **SLO 健康度透出**：从"阈值告警"升级为"SLO 健康度评分"——用户看到的不是"延迟告警触发"，而是"当前推理服务健康度 62 分，主要劣化因子：显存压力 + 队列积压，预计影响时间窗口：XX"。

**目标态：可观测模块从"数据展示层"升级为"平台感知层"，成为 FOAI 智能能力的统一数据入口。**

## P5 FOAI · 记忆沉淀

### 为什么记忆重要？

当前，FOAI 的使用经验大量沉淀在**人**而非**平台**上：

- 某模型在高并发下需要调整哪些参数，靠老用户口耳相传
- 某类报错的处理路径，靠排查过的工程师记得
- 某个服务的最佳配置组合，靠试错后写进 Wiki

这导致：**每个新用户都在重复踩同样的坑；平台本身没有在变聪明。**

参考 Anthropic 对 Agent 系统的定义——真正有价值的 Agent 系统需要具备 **Memory（记忆）** 能力，以跨交互边界地积累和复用知识。这一原则同样适用于推理平台本身。

### FOAI 可以沉淀哪些记忆？

```
┌──────────────────────────────────────────────────────┐
│                  FOAI 记忆资产分类                     │
│                                                      │
│  短期记忆（Session）         长期记忆（Platform）       │
│  ─────────────────         ──────────────────────   │
│  · 当前会话上下文            · 黄金配置模板 (Golden Path) │
│  · 本次部署参数              · 历史故障根因库             │
│  · 当前服务状态快照           · 模型性能基线（Benchmark）  │
│                             · 用户操作行为模式           │
│                             · 已验证的排障路径           │
└──────────────────────────────────────────────────────┘
```

**重点方向：长期记忆的平台化**

| 记忆类型 | 沉淀方式 | 复用场景 |
|---|---|---|
| **黄金配置模板（Golden Path）** | 从历史成功部署中提炼，结构化存储 | 新用户部署时推荐最优起点 |
| **故障根因库** | 每次告警闭环后自动写入，LLM结构化摘要 | 下次相似故障秒级归因 |
| **模型性能基线** | 模型评估结果自动归档 | 部署前性能预期透出，避免上线后"性能不达预期"投诉 |
| **排障路径图谱** | 从工单+操作日志中挖掘成功路径 | 引导用户按最优路径自助排障 |

### 记忆沉淀带来的系统性变化

- **用户行为变化**：从"我来问平台"变成"平台来告诉我"——新用户的学习曲线大幅压缩，有经验用户的决策效率提升
- **系统行为变化**：平台每经历一次故障、一次部署，都在变聪明——知识不再随人员流动而流失
- **L3 的前提**：记忆沉淀是意图驱动的基础。没有足够的历史经验作为"先验知识"，LLM 无法做出可靠的自动推导

## P6 FOAI · 智能决策

### AI Copilot 的定位：不是界面，是能力层

AI Copilot 不应被理解为"加了个聊天框"，而是 FOAI 平台的**智能决策能力层**——它连接用户意图与平台操作，驱动 L2→L3 演进。

```
用户意图（自然语言/结构化）
        ↓
   AI Copilot（决策层）
   · 意图解析
   · 上下文整合（调用可观测层 + 记忆层）
   · 方案推导与风险评估
   · 操作执行 / 引导确认
        ↓
   FOAI 平台操作（部署/配置/诊断/扩缩容）
```

### 三个核心落地场景

**① 部署决策辅助（L2 → L3）**

用户描述需求："我要部署一个支持长上下文的代码补全模型，并发 50，延迟要求 P99 < 2s"

Copilot 行为：
- 从 Golden Path 库匹配最近似的成功案例
- 结合当前集群资源状态，推导推荐配置（实例规格、并发数、KV Cache策略）
- 提示潜在风险（如：长上下文模型在当前显存配置下 KV Cache 压力较大）
- 生成可直接提交的部署草稿，用户确认后一键执行

**② 异常归因与处置引导（L2）**

触发条件：P99 延迟 SLO 劣化告警

Copilot 行为：
- 自动拉取近 30 分钟拓扑快照 + 指标趋势 + 变更记录
- 结合故障根因库，输出结构化归因：
  > "根因大概率为：并发请求激增（+240%）导致 KV Cache 淘汰率上升，队列深度达 142。建议操作：①临时限流至当前 60% QPS ②扩容 1 个推理实例。"
- 提供一键操作入口，确认后直接执行，无需用户跳转多个页面

**③ 模型选型决策（L2）**

用户需求："我需要一个适合法律文本摘要的模型，输入长度 8K 左右，对准确性要求高"

Copilot 行为：
- 检索模型广场，结合模型评估基线数据（摘要任务评分、长文本支持、延迟表现）
- 输出结构化推荐报告，列出 Top 3 候选及差异对比
- 引导用户进入模型体验 / 竞技场进行对比验证

### 技术实现路径

| 能力模块 | 实现方式 |
|---|---|
| 意图解析 | LLM + Prompt Engineering，结合平台 Schema 约束输出 |
| 上下文整合 | MCP Server 化（可观测数据、记忆库、平台状态） |
| 方案推导 | RAG（检索 Golden Path + 故障库） + LLM 推理 |
| 操作执行 | Tool Calling，调用 FOAI 标准 API |
| 风险拦截 | 结构化校验 + 规则引擎兜底（防止 LLM 幻觉直接操作） |

**核心设计原则（参考 Anthropic Writing Effective Tools for AI Agents）：工具定义要精确、错误反馈要结构化、操作要可撤销、高风险操作必须人工确认。**

## P7 FOAI · 从用户易用到 Agent 易用

### 为什么要关注 Agent 易用？

FOAI 的用户正在发生结构性变化：**越来越多的调用不来自人，而来自 Agent。**

当一个 Agent 调用 FOAI 的推理服务时，它需要的不是"好看的 UI"，而是：
- **精确的工具定义**：能正确理解接口语义，不产生歧义
- **结构化的错误反馈**：报错信息能被 LLM 解析，并触发正确的重试或上报
- **丰富的状态透出**：让 Agent 知道当前服务状态，能做出合理的调度决策
- **可预测的行为**：相同输入产生稳定输出，Agent 能够建立可靠的期望

### ACI 设计原则

ACI（Agent-Computer Interface）是面向 AI Agent 的接口设计范式，是 HCI（Human-Computer Interface）在 Agent 时代的延伸。

参考 Anthropic《Writing Effective Tools for AI Agents》（2025）提出的核心原则：

| 设计维度 | 人类界面（HCI）习惯 | Agent 接口（ACI）要求 |
|---|---|---|
| **错误信息** | 提示用户"操作失败，请重试" | 结构化错误码 + 根因 + 建议操作（可被LLM解析） |
| **接口描述** | 字段名 + 类型注释 | 丰富的语义描述，包含典型用法、边界条件、副作用 |
| **状态返回** | HTTP 200 / 500 | 细粒度状态码 + 上下文信息（当前队列、资源状态） |
| **幂等性** | 尽量保证 | 强制保证，Agent 重试时不能产生副作用 |
| **操作粒度** | 组合操作（批量） | 原子操作为主，便于 Agent 精确控制 |

### FOAI 的 ACI 落地方向

**① API 错误结构化（近期可落地）**

当前：推理服务返回 `500 Internal Server Error`，Agent 无法判断是配置问题、资源问题还是模型问题。

目标：返回结构化错误体：
```json
{
  "error_code": "INFERENCE_OOM",
  "error_message": "GPU显存不足，当前占用97%，请求需要额外2.3GB",
  "suggested_action": "reduce_batch_size_or_scale_out",
  "retryable": false,
  "context": {
    "current_memory_usage": "97%",
    "required_additional_memory": "2.3GB"
  }
}
```

**② MCP Server 化（与架构师方向对齐）**

将 FOAI 核心能力封装为 MCP Server，包括：
- 推理服务状态查询工具
- 部署操作工具（支持参数校验 + 风险提示）
- 模型评估结果查询工具
- 告警与健康度查询工具

Agent 通过标准 MCP 协议调用，无需关心 FOAI 内部实现细节。

**③ 服务可观测性对 Agent 透出**

Agent 调度推理服务时，能实时获取：当前 QPS、队列深度、预估等待时间、健康度评分。Agent 可据此做出动态路由、降级、重试等决策——而非盲目调用。

### 价值

- **对内**：FOAI 进入 AI Native 工作流，不再只是"被人点击的平台"，而是"可被 Agent 编排的能力节点"
- **对外**：在企业级 Agent 应用爆发的背景下，ACI 友好是 FOAI 作为 MaaS 平台的核心差异化能力之一
- **与架构师方向的呼应**：MCP Server 化 + 双轨交互（轨道 B AI Native）是同一个演进方向的不同切面

---

## P8 总结：三个核心判断

**判断一：易用性的竞争已经分层**

L1（可操作）是入场券，不是竞争力。真正的分化在 L2（可理解）和 L3（意图驱动）。FOAI 2026 的核心任务是在 L2 做深，同时在 L3 建立技术可信度。

**判断二：可观测性 + 记忆沉淀是所有智能能力的基础设施**

没有结构化的感知层，AI Copilot 就是无根之木。2026 年最值得投入的基础工作是：把现有观测数据语义化、把历史经验资产化。这两件事做好了，智能决策和 Agent 易用才有可靠的数据底座。

**判断三：Agent 易用不是锦上添花，是方向性押注**

企业级 AI 应用的核心形态正在从"人使用工具"转向"Agent 编排工具"。FOAI 如果只对人友好，将逐渐被排除在 Agent 工作流之外。ACI 设计、MCP Server 化、结构化错误反馈——这些投入的回报周期比预期的更短。
