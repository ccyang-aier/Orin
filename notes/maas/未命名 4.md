# FOAI · 易用性竞争力
## 落地进展与持续演进方向

> **研讨材料 · 2026.04**

---

## P1｜破题：易用，不是体验修补，是 FOAI 在 Agent 时代的主竞争力

> **核心判断：易用性的竞争，已经不是 UI 好不好看，而是平台能不能把 AI 的复杂性吸收掉，把正确结果更稳定地交给用户。**

AI 平台的能力已经越来越强，但用户真正敢把业务交给平台的，还是少数。

原因不在于功能缺失，而在于：

- 用户每一步都不确定"这样配对不对"
- 出了问题不知道为什么、不知道该怎么办
- 评估了一堆模型，但最终还是靠猜
- 平台每个模块都能用，但连起来走一遍却很费劲

**易用，是"永快、永新、体验好"真正交付到用户手里的最后一公里。**

---

## P2｜趋势判断：AI 平台易用性的竞争主战场已经迁移

> **核心判断：从"降低页面操作门槛"，升级到"吸收 AI Infra / Agent Infra 的系统复杂性"。**

### 三个迁移正在发生

| 过去的竞争重点     | 正在成为新重点     |
| ----------- | ----------- |
| 页面操作简单、引导清晰 | 系统行为可理解、可解释 |
| 功能全、模块多     | 意图可表达、决策有依据 |
| 看指标、看日志     | 给原因、给建议、给动作 |

### 行业趋势支撑

1. **Agent 时代，难点从"生成答案"转向"约束行为"**
   记忆、工具调用、执行轨迹、权限边界、异常恢复，正在成为 AI Infra 的核心复杂性。

2. **Memory 从缓存能力升级为系统能力**
   分层记忆、长期记忆、经验沉淀，正在成为 Agent 基础设施的标配。

3. **Observability 从"看指标"升级为"解释系统为什么这么做"**
   TTFT、吞吐、GPU 利用率只是开始；串联配置、上下文、执行轨迹、异常原因才是终局。

4. **用户的不安，本质是"心智模型不匹配"**
   不知道系统记住了什么、为什么这么做、哪里能改——这直接影响信任与易用性。

---

## P3｜重新定义：FOAI 的"易用"是什么

> **核心判断：FOAI 的易用 = 低摩擦 + 低认知负担 + 低试错成本 + 高结果确定性。**

不是"页面更简单"，而是：

```
用户能清楚地表达意图
   ↓
平台能可靠地理解并执行
   ↓
结果可解释、可验证、可信任
   ↓
经验可沉淀、系统越用越聪明
```

### 易用的四个层次

| 层次 | 定义 | 对应目标 |
|---|---|---|
| **会用** | 上手不费劲 | 传统易用，降低操作门槛 |
| **看得懂** | 系统不是黑盒 | 可理解：行为可解释，状态可感知 |
| **可托付** | 平台能替用户做合理决策 | 意图驱动：不靠猜，有依据 |
| **可进化** | 平台沉淀经验，越用越懂场景 | 经验记忆：成功路径变平台能力 |

**当前 FOAI 在"会用"层有基础，在"看得懂、可托付、可进化"层存在明显断裂。**

## P4｜现状诊断：不是功能不够，是存在四个断裂

> **核心判断：FOAI 当前的易用性问题，本质是四个断裂，而不是功能缺失。**

### 四个断裂

```
┌─────────────────────────────────────────────┐
│  认知断裂                                    │
│  用户看到慢、看到错，但不知道为什么           │
│  配置参数没有语义，排障路径靠经验积累         │
└─────────────────────────────────────────────┘

┌─────────────────────────────────────────────┐
│  决策断裂                                    │
│  评估、体验、部署、运行是四个孤岛             │
│  每次部署决策，用户还是在"靠猜"              │
└─────────────────────────────────────────────┘

┌─────────────────────────────────────────────┐
│  处置断裂                                    │
│  看到异常到行动之间，有大量手动跳转           │
│  告警触发了，但没有处置路径                  │
└─────────────────────────────────────────────┘

┌─────────────────────────────────────────────┐
│  进化断裂                                    │
│  成功经验、踩坑案例、最佳配置无法沉淀         │
│  每个新用户/新场景都要从零开始摸索           │
└─────────────────────────────────────────────┘
```

> 来源依据：AI4RD 用户真实反馈，高频词：错误信息不可读、任务堆积无感知、配置黑盒、告警混乱、跨模块割裂。

---

## P5｜方向一：从"部署工具"升级为"意图编译器"

> **核心判断：部署易用的下一阶段，不是模板更多，而是让用户表达目标，平台负责编译成方案。**

### 当前问题

用户手配几十个底层参数（并发数、副本数、资源规格、超时时间……），
每个参数背后的含义和影响，大多数用户并不清楚。

### 升级方向

```
【当前模式】
用户 → 手配参数 → 提交部署 → 希望没配错

【意图编译器模式】
用户表达意图："这个服务用于在线客服，预计并发 50，要求响应 <2s"
      ↓
平台推理出：推荐规格 / 预估资源消耗 / 风险评分 / 多方案对比
      ↓
用户确认 → 部署 → 平台持续监控是否符合意图
```

### 关键能力建设

- **场景意图式部署**：用业务语言描述诉求，而非技术参数
- **部署前风险评分**：配置合理性预检，不是等到失败才报错
- **多方案对比推荐**：最低成本 / 最优性能 / 最稳妥，用户自主选择
- **历史成功经验沉淀**：把已验证的配置路径，变成下次部署的推荐起点

---

## P6｜方向二：从"监控平台"升级为"诊断副驾"

> **核心判断：观测的终局不是看板更多，而是系统直接给原因、建议和动作。**

### 当前问题

指标有了、拓扑有了、告警有了——但用户看完还是不知道该怎么办。

"慢"只是现象，根因可能是：上下文超长、显存碎片、资源竞争、配置不当、模型本身……

### 升级方向

```
【当前模式】
异常发生 → 看告警 → 看指标 → 看日志 → 猜原因 → 尝试处理

【诊断副驾模式】
异常发生 → 系统自动语义化分析 → 输出因果诊断链
        → 给出处置建议 → 支持一键进入处置闭环
```

### 关键能力建设

- **"慢"的语义拆解**：把延迟分解为 prefill / decode / 排队 / 网络等子项，直接定位瓶颈
- **故障因果图**：事件链路可视化，而不是指标堆砌
- **告警语义分级**：区分"需要立即处理"与"观察即可"，减少告警噪音
- **从告警/拓扑一键进入处置闭环**：看到问题 → 理解原因 → 采取行动，三步不断链
- **建立平台级"故障语义层"**：把底层错误码统一映射为用户能理解、能判断、能行动的故障语言

---

## P7｜方向三：从"模型货架"升级为"模型决策系统"

> **核心判断：模型生命周期的关键，不是模型更多，而是部署决策不靠猜。**

### 当前问题

评估完了，不知道该选哪个；选了部署上去，不知道线上跑得好不好；
模型广场是模型广场，评估是评估，部署是部署——三个孤岛。

### 升级方向

```
业务任务诉求
   ↓
模型推荐（按任务类型，而不是按模型名）
   ↓
体验 / 评估（Arena + 自定义基准）
   ↓
评估结论 → 直达部署决策（一键部署、带评估报告）
   ↓
线上运行数据（延迟、质量、成本）→ 反哺离线评估基准
   ↓
形成闭环：评估越来越准，决策越来越有依据
```

### 关键能力建设

- **按任务选模型**：用业务场景描述诉求，而非靠人工对比模型名称
- **评估结果直达部署动作**：评估页直接出部署建议，不再手动转抄参数
- **线上运行数据反哺离线评估**：在线真实表现回写到评估基准
- **模型使用说明书**：每个模型的能力边界、适用场景、已知局限——让用户"用前知边界"

---

## P8｜方向四：从"固定 WebUI"升级为"人机双轨操作面"

> **核心判断：统一交互的终局，不是术语统一、风格一致，而是 GUI、自然语言、Agent 调用共享同一套操作原子。**

### 当前问题

- 不同模块：术语不同、层级不同、跳转不连贯
- 操作只有 GUI 一个通道：不支持自然语言指令，不支持 Agent 自动化调用
- 随着 Agent 大规模进入生产环境，人工操作面与机器操作面将越来越割裂

### 升级方向

```
【当前】
  GUI 操作 ──────────────────────→ 平台执行

【人机双轨操作面】
  GUI 操作  ─────────────┐
  自然语言指令  ──────────┤→ 统一操作语义层 → 平台执行
  Agent API 调用  ────────┘
```

### 关键能力建设

- **第一步（近期）**：统一术语体系 + 资源层级 + 跨模块跳转一致性——建立"认知地图"
- **第二步（中期）**：操作原子化，GUI 动作与 API 动作语义对齐；支持自然语言触发常见操作
- **第三步（远期）**：面向 Agent 时代，平台操作面可被 Agent 调用；按任务动态生成操作界面

> **前瞻判断**：当 FOAI 成为 Agent Infra 的一部分，"操作平台的"可能不再是人，而是另一个 Agent。今天建立统一操作语义层，是为明天做地基。

## P9｜里程碑：未来一年，三个可交付的演进目标

> **核心判断：易用性不是一次性优化，而是一个平台能力的持续跃迁。我们要让 FOAI 在一年内完成从"可用"到"可信任"的跨越。**

### 三个里程碑

```
┌─────────────────────┐    ┌─────────────────────┐    ┌─────────────────────┐
│   M1：可理解         │    │   M2：可托付         │    │   M3：可进化         │
│                     │    │                     │    │                     │
│  系统行为不再是黑盒  │ →  │  平台参与用户决策    │ →  │  经验沉淀为系统能力  │
│                     │    │                     │    │                     │
│  · 故障语义层上线   │    │  · 意图式部署        │    │  · 历史配置推荐      │
│  · 诊断副驾 v1      │    │  · 部署风险评分      │    │  · 在线数据反哺评估  │
│  · 认知地图统一     │    │  · 模型决策系统      │    │  · 经验知识库        │
└─────────────────────┘    └─────────────────────┘    └─────────────────────┘
      H1 2026                    H2 2026                   2027 方向
```

### 收束

> **易用，不是把页面做简单，而是把复杂性吃掉，把正确结果更稳定地交付给用户。**
>
> 从"会用平台"，到"敢把业务交给平台"——这是 FOAI 易用性演进的核心命题。
