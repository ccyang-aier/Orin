# 3. 如何用 Codex 设计精美且炫酷的 UI

这篇教程解决一个具体问题：当你想让Codex不只是“写一个能跑的页面”，而是产出更像产品级作品的 UI、动效、演示稿、视频、Agent 前端或可复用设计系统时，应该如何去做。

需要先明确一个边界：Codex 的“插件”和“技能”不是同一个层级。插件是可安装的能力包，通常由 `.codex-plugin/plugin.json`、`skills/`、`.app.json`、`assets/` 等组成；技能是插件或本地目录中的 `SKILL.md` 工作流说明，用来告诉 Codex 在特定任务中如何行动。OpenAI 官方文档也把 Web development、Figma-to-code、响应式前端、UI 细节迭代、视觉验证等列为 Codex 的典型用例，但可用插件会受当前账号、工作区 Apps 控制、安装状态和本地环境影响。

## 1. 先建立UI任务的分层地图

不要把所有UI任务都交给同一个插件。更稳的方式是先判断任务属于哪一层：

| 层级     | 典型任务                               | 优先选择                                 |
| ------ | ---------------------------------- | ------------------------------------ |
| 视觉探索   | 海报、封面、缩略图、品牌方向、灵感图                 | `imagegen`、Canva、Figma、BioRender     |
| 产品界面设计 | Web/App 页面、组件、设计系统、Figma mockup    | Figma、Browser、Chrome、GitHub          |
| 前端实现   | React/Next.js 页面、Agent 聊天 UI、响应式布局 | Codex 基础能力、Browser、Chrome、外部 UI 项目   |
| 动效与视频  | 产品宣传片、动画标题、交互演示、字幕、音频可视化           | HyperFrames、Remotion、HeyGen          |
| 专业图形   | 科研流程图、机制图、学术图示、复杂知识图               | BioRender、Figma FigJam、Presentations |
| 验证与迭代  | 截图对比、移动端/桌面端检查、可访问性、文字溢出           | Browser、Chrome、Figma、Playwright 类测试  |

一个实用原则是：先用设计插件把“该长什么样”定清楚，再让Codex写代码，最后用浏览器插件做视觉验收。直接从空白代码开始做炫酷UI，容易得到“功能可用但审美普通”的结果。

## 2. Codex内置UI设计能力

本节基于当前环境中可见的插件缓存和技能列表整理。它不等于 Codex 全部可安装插件；例如 OpenAI 官方用例页提到 React Native / Expo 相关专用插件场景，但当前本机插件缓存中没有对应插件，因此本文只把它作为可关注方向，不展开具体用法。

### 2.1 插件结构怎么看

在本机缓存里，一个插件通常长这样：

```text
<plugin-name>/<version-or-hash>/
  .codex-plugin/plugin.json
  .app.json
  assets/
  skills/
    <skill-name>/SKILL.md
```

其中：
- `plugin.json`：插件元数据，包含名称、版本、介绍、作者、官网、能力类别、默认提示词、图标和是否包含 skills/apps；
- `.app.json`：连接外部 App 或 MCP 工具的声明，决定 Codex 能否实际调用 Canva、Figma、GitHub、HeyGen 等工具；
- `skills/*/SKILL.md`：具体工作流，例如“生成 Figma 页面”“创建 Canva 品牌演示稿”“用 HyperFrames 渲染视频”；
- `assets/`：图标、Logo、插件展示资源；
- 有些插件只有 App 工具而没有技能，例如当前 BioRender 插件只有搜索类 App 能力，没有本地 `skills/` 目录。

### 2.2 Figma：从设计系统到页面 mockup 的主力

**结构组成：**

当前 Figma 插件包含 `.codex-plugin/plugin.json`、`.app.json`、`assets/` 和 8 个技能：

- `figma-use`：所有 `use_figma` 写操作或复杂读操作的强制前置技能；
- `figma-create-new-file`：创建新 Figma / FigJam / Slides 文件前的强制前置技能；
- `figma-generate-design`：把页面、弹窗、抽屉、侧边栏等多区块 UI 写入 Figma；
- `figma-generate-library`：从代码库构建设计系统，包含变量、tokens、组件、主题和文档页；
- `figma-code-connect`：创建或维护 Figma Code Connect 映射，让设计组件和代码组件互相解释；
- `figma-generate-diagram`：在 FigJam 中生成流程图、架构图、时序图、ERD、状态图等；
- `figma-use-figjam`：FigJam 场景下使用 `use_figma` 的补充规则；
- `figma-use-slides`：Figma Slides 场景下使用 `use_figma` 的补充规则。

**适用场景：**

- 需要把代码页面同步为 Figma 画稿；
- 需要先做产品界面 mockup，再让 Codex 实现；
- 需要从现有代码抽取 design tokens、组件库和变量；
- 需要设计系统，而不只是一次性页面；
- 需要流程图、系统图、架构图、用户路径图。

**使用方式：**

1. 先让 Codex 读取目标页面代码、截图或需求，明确主要区块、组件、状态和断点；
2. 如果目标是 Figma 页面，让 Codex 使用 `figma-generate-design` + `figma-use`；
3. 如果目标是设计系统，让 Codex 使用 `figma-generate-library` + `figma-use`，并按 discovery、foundations、file structure、components、QA 分阶段推进；
4. 如果目标是图表或架构说明，让 Codex 使用 `figma-generate-diagram`；
5. 生成后要求 Codex 对照截图或设计目标检查间距、字体、颜色、图片、响应式状态。

**推荐提示词：**

```text
请使用 Figma 插件，把当前 Next.js 页面整理成一个可编辑的 Figma screen。
先读取代码和现有设计 token，列出页面区块、组件、状态和需要复用的设计系统资源；
再创建 Figma 页面，最后给我一份设计差异清单。
```

### 2.3 Canva：营销设计、社媒物料、品牌演示稿

**结构组成：**

当前 Canva 插件包含 App 工具和 3 个技能：

- `canva-branded-presentation`：从 brief、outline、Canva doc 或设计链接生成品牌演示稿；
- `canva-resize-for-all-social-media`：把一个设计适配到多个社媒尺寸；
- `canva-translate-design`：翻译 Canva 设计文本，同时尽量保留原布局。

App 工具还支持搜索设计、读取设计内容、生成设计候选、从候选创建设计、上传素材、开始编辑事务、执行编辑操作、提交编辑事务等。

**适用场景：**

- 需要把产品、项目、教程做成演示稿；
- 需要制作封面、海报、社媒图、YouTube 缩略图、报告、proposal；
- 需要把已有视觉设计翻译成另一种语言；
- 需要从一个主视觉批量生成多平台尺寸。

**使用方式：**

1. 先给 Codex 明确内容源：文本 brief、Markdown、已有 Canva 链接、图片素材或品牌规范；
2. 如果是演示稿，先让 Codex 生成 slide plan，再进入 Canva 的候选生成流程；
3. 如果是编辑已有设计，必须先定位设计 ID，再进入编辑事务，执行修改，最后提交事务；
4. 如果是社媒分发，先确认主设计，再执行多尺寸 resize；
5. 每次生成后让 Codex 返回可编辑 Canva 链接和关键设计方向。

**推荐提示词：**

```text
请用 Canva 插件把这篇教程生成一份 12 页品牌演示稿。
先给出 slide plan：每页标题、目标、要点和视觉建议；
确认结构后再生成候选，并保留可编辑链接。
```

### 2.4 BioRender：科研、医学、生物图示

**结构组成：**

当前 BioRender 插件包含 `.codex-plugin/plugin.json`、`.app.json` 和图标资源，没有本地技能目录。可用能力是搜索 BioRender 的科学模板库，返回模板标题、描述和可跳转的详情链接。

**适用场景：**

- 科研机制图、实验流程图、通路图；
- 生物医学插图、论文/基金申请图；
- 需要科学准确图标和模板，而不是普通视觉插画。

**使用方式：**

1. 描述领域、对象和图示类型，例如 “GLP-1 pathway template”；
2. 让 Codex 用 BioRender 搜索模板；
3. 从返回结果中挑选模板；
4. 在 BioRender 应用里继续编辑和导出；
5. 如果要写入教程或论文，保留模板来源与授权边界。

**推荐提示词：**

```text
请用 BioRender 搜索 5 个适合解释 RAG 检索流程的科学/技术风格模板，
返回模板标题、适用场景和可打开的链接。
```

### 2.5 HyperFrames by HeyGen：HTML + GSAP 的高质量视频工作流

**结构组成：**

当前 HyperFrames 插件包含 5 个技能：

- `hyperframes`：HTML 视频 composition 的核心工作流；
- `hyperframes-cli`：初始化、预览、渲染、转写、TTS、诊断、检查等 CLI 工作流；
- `gsap`：GSAP 动画参考；
- `hyperframes-registry`：安装和接入 HyperFrames registry blocks/components；
- `website-to-hyperframes`：把网站捕获并转成产品宣传视频。

**适用场景：**

- 产品宣传片、教程片头、社媒短视频；
- 网站转视频、产品 tour、版本发布视频；
- 音频反应式动画、字幕、高级转场、动态图表；
- 需要 HTML/CSS 可控性，又要导出视频。

**使用方式：**

1. 先定义视觉风格，不要直接写默认色和默认字体；
2. 先做静态 hero frame，确认布局不会溢出；
3. 再用 GSAP 添加入场、强调、退出动画；
4. 用 HyperFrames CLI 做 lint、inspect、preview；
5. 最后 render 输出视频。

**推荐提示词：**

```text
请用 HyperFrames 为这个产品页面做一个 20 秒发布视频。
先定义 DESIGN.md：色彩、字体、运动风格和禁用项；
然后写 HTML composition，使用 GSAP 做分镜动效，并通过 preview/render 验证。
```

### 2.6 Remotion：React 程序化视频与动态图形

**结构组成：**

当前 Remotion 插件主要暴露 `remotion-best-practices` 技能。它覆盖 Remotion 新项目初始化、Studio 预览、单帧渲染检查，并指向 3D、动画、音频、字幕、图表、Lottie、地图、文字动画、转场、透明视频等规则文件。

**适用场景：**

- 你已经在 React/Next.js 技术栈里；
- 需要用代码生成可重复渲染的视频；
- 需要动态图表、字幕、音频可视化、模板化视频；
- 需要把数据驱动的内容批量生成视频。

**使用方式：**

1. 新项目可用 `npx create-video@latest --yes --blank --no-tailwind my-video`；
2. 使用 `npx remotion studio` 预览；
3. 对重要帧用 `npx remotion still [composition-id] --scale=0.25 --frame=30` 检查；
4. 涉及字幕、音频、3D、图表时，要求 Codex 读取对应规则文件；
5. 让 Codex 把视频组件拆成 composition、sequence、asset、timing、theme 几层。

**推荐提示词：**

```text
请用 Remotion 做一个 30 秒产品功能发布视频。
使用 React 组件组织分镜，包含动态图表、字幕和 CTA；
先生成 composition 结构，再做单帧渲染检查。
```

### 2.7 HeyGen：Avatar、真人口播、产品讲解视频

**结构组成：**

当前 HeyGen 插件包含 2 个技能：

- `heygen-avatar`：创建可复用的头像/声音身份；
- `heygen-video`：使用 v3 Video Agent pipeline 生成 presenter 视频。

App 工具支持头像同意流程、从图片生成视频、读取视频资源等。技能中特别强调不要直接调用过时 v1/v2 REST 端点，而应走 HeyGen App 或 HeyGen CLI。

**适用场景：**

- 需要真人感讲解、课程引导、团队更新、销售视频；
- 需要把产品 demo 包装成“有人讲”的视频；
- 需要长期复用同一个数字人身份。

**使用方式：**

1. 如果需要固定数字人，先用 `heygen-avatar` 建立 avatar；
2. 再用 `heygen-video` 生成口播视频；
3. 明确视频目标、受众、时长、语言、横竖屏、素材和语气；
4. 不要把 API payload、session id 等内部信息写入最终文案；
5. 结果交付以视频链接、缩略图、时长和 avatar 信息为主。

**推荐提示词：**

```text
请用 HeyGen 做一个 45 秒中文产品讲解视频。
目标受众是第一次了解这个工具的开发者，语气专业但不生硬；
如果当前没有 avatar，请先创建一个可长期复用的 presenter 身份。
```

### 2.8 Browser 与 Chrome：前端 UI 的真实验收层

**结构组成：**

当前环境有两个浏览器相关插件：

- `Browser`：Codex 内置浏览器，适合 localhost、file、当前 in-app browser tab、网站截图和交互验证；
- `Chrome`：连接用户 Chrome，适合需要登录态、Cookie、扩展、已有标签页或远程 authenticated 页面。

**适用场景：**

- 验证本地页面是否渲染；
- 检查桌面端/移动端布局；
- 捕获截图并和设计稿对比；
- 点击、输入、打开弹窗、跑关键用户路径；
- 验证文本是否溢出、图片是否加载、动效是否阻塞。

**使用方式：**

1. 写完前端后启动 dev server；
2. 用 Browser 打开 `localhost`；
3. 对关键 viewport 截图；
4. 检查 UI 是否空白、错位、遮挡、文字溢出；
5. 如果页面依赖用户账号或真实浏览器状态，再切到 Chrome。

**推荐提示词：**

```text
请实现这个页面后用 Browser 做视觉验收：
检查 1440px 桌面、390px 移动端；
确认首屏不空白、文字不溢出、按钮可点击、图片加载正常，并给出截图结论。
```

### 2.9 Presentations、Documents、Spreadsheets：视觉文档的辅助插件

这些插件不专门做产品 UI，但在“把 UI 方案表达清楚”时很有用：

- `Presentations`：生成和验证 PowerPoint / Google Slides 风格的高质量分析型 deck；
- `Documents`：生成、编辑、渲染和检查 `.docx`；
- `Spreadsheets`：生成、分析、可视化 `.xlsx`、`.csv`、`.tsv`。

适合做设计方案汇报、竞品分析、组件清单、视觉规范、数据图表和项目复盘。它们的价值不是替代 Figma，而是把设计思路、数据和方案沉淀成可传阅资产。

### 2.10 imagegen：视觉探索和位图素材

`imagegen` 是系统级技能，适合生成或编辑位图资产，例如照片、插画、贴图、精灵图、mockup 背景、透明抠图。它不适合替代 SVG、代码组件或设计系统，但非常适合：

- hero 背景图；
- 产品概念图；
- 插画风格探索；
- 视频封面；
- 游戏或角色资产；
- UI 中的局部位图素材。

使用时要明确尺寸、风格、主体、构图、色彩、透明背景需求，以及后续会在哪里使用。

## 3. 外部高质量项目：把 Agent UI 和前端开发接入 Codex

这一部分借助 GitHub 插件搜索，并结合当前可访问的项目 README / 文档整理。筛选标准是：

- 近期仍有维护迹象或 GitHub 页面显示活跃；
- 能与 Codex 的代码编辑、浏览器验证、Figma/Canva/视频插件形成工作流；
- 不是只能在线使用的封闭工具；
- 能落地到 React / Next.js / shadcn / Agent 协议 / 前端组件；
- 排除明显偏通用聊天、维护弱、或难以接入 Codex 本地项目的方案。

### 3.1 CopilotKit：Agentic 应用和 Generative UI 的前端栈

**项目定位：**

CopilotKit 是面向 full-stack agentic applications、Generative UI 和 chat applications 的 SDK，支持 React 与 Angular，也是 AG-UI Protocol 背后的核心项目之一。

**结构组成：**

- 前端 UI 组件：如 copilot chat、sidebar、popup、textarea 等；
- runtime / backend 连接层：把前端状态、用户动作、agent 输出连接起来；
- Generative UI：让 agent 不只返回文本，还能返回 UI；
- AG-UI 生态：把 agent 事件流标准化，方便不同后端 agent 接入前端。

**适用场景：**

- SaaS 产品内嵌 AI Copilot；
- 需要 agent 读取当前页面状态并触发应用动作；
- 需要 tool call、人类确认、生成式 UI；
- 想把 AI 助手做成产品能力，而不是独立聊天窗口。

**Codex 集成方式：**

1. 让 Codex 先阅读现有 React/Next.js 页面和状态管理；
2. 选择接入位置：侧边栏、浮窗、输入框增强、页面内 Copilot；
3. 安装 CopilotKit 包并接入 provider；
4. 定义可暴露给 agent 的 actions；
5. 用 Browser 验证用户路径和 UI 状态；
6. 若需要标准化 agent 事件流，再引入 AG-UI。

### 3.2 AG-UI：Agent 到用户界面的事件协议

**项目定位：**

AG-UI 是开放、轻量、基于事件的 Agent-User Interaction Protocol，用来标准化 AI agent 与用户界面的连接。它和 MCP、A2A 的分工不同：MCP 偏 agent 到工具，A2A 偏 agent 到 agent，AG-UI 偏 agent 到用户界面。

**结构组成：**

- 标准事件类型：文本消息、工具调用、状态更新、中间输出等；
- transport 适配：SSE、WebSocket、webhook 等；
- reference HTTP implementation；
- connector 和 framework integrations。

**适用场景：**

- 后端 agent 框架不统一，但前端想统一渲染；
- 需要实时展示 agent 的思考、工具调用、审批和结果；
- 需要把 LangGraph、CrewAI、PydanticAI、Mastra 等后端接到同一套 UI；
- 需要为 Codex 生成的前端保留协议层扩展性。

**Codex 集成方式：**

1. 先决定前端项目使用 CopilotKit、assistant-ui、AI Elements 还是自研组件；
2. 把后端 agent 输出翻译成 AG-UI 事件；
3. 前端用协议事件驱动 message、tool-call、approval、status 组件；
4. 用 Browser 验证长任务、断线、重试和取消状态；
5. 用 Figma 把不同 agent 状态画成组件变体。

### 3.3 assistant-ui：生产级 AI Chat 组件库

**项目定位：**

assistant-ui 是开源 TypeScript/React 库，用于快速构建生产级 AI chat 体验。它提供 composable primitives，而不是单一黑盒 widget，并支持 Vercel AI SDK、LangGraph、AG-UI、A2A、自定义 data-stream backend 等。

**结构组成：**

- 核心组件：`Thread`、`Message`、`Composer`、`ThreadList`、`ActionBar` 等；
- runtime provider：把 UI 与 AI SDK / LangGraph / 自定义后端连接；
- shadcn 风格组件：可复制到项目内继续定制；
- Generative UI：支持把 tool calls / JSON 渲染为 React 组件，支持 inline human approvals；
- 可选 assistant-cloud：托管 thread history、telemetry、文件存储。

**适用场景：**

- 想快速做一个 ChatGPT-like UI；
- 需要 streaming、auto-scroll、附件、markdown、代码高亮、键盘快捷键和可访问性；
- 需要兼容多种 agent runtime；
- 不想从零处理复杂聊天交互细节。

**Codex 集成方式：**

```bash
npx assistant-ui@latest create
npx assistant-ui@latest init
npm install @assistant-ui/react @assistant-ui/react-ai-sdk
```

然后让 Codex：

1. 把组件复制到当前项目；
2. 接入 `AssistantRuntimeProvider`；
3. 根据产品风格改写 shadcn/Tailwind 样式；
4. 加入 tool call、approval、附件和错误状态；
5. 用 Browser 检查 streaming、滚动、移动端输入框和代码块。

### 3.4 Vercel AI Elements：AI-native shadcn 组件注册表

**项目定位：**

AI Elements 是 Vercel 基于 shadcn/ui 的 AI 原生组件库和 registry，提供 conversations、messages、code blocks、reasoning displays 等可复制组件。

**结构组成：**

- CLI：`npx ai-elements@latest`；
- shadcn registry：可通过 URL 安装全部或单个组件；
- 组件目录：安装后进入项目的 `components/ai-elements/`；
- 典型组件：conversation、message、response、code-block、reasoning 等；
- 与 AI SDK / Next.js / Tailwind CSS Variables 模式配套。

**适用场景：**

- 项目已经使用 Next.js、AI SDK、shadcn/ui、Tailwind；
- 需要可完全掌控源码的 AI UI 组件；
- 需要 reasoning、tool output、code block、message thread 等 AI 专用 UI；
- 想让 Codex 在本地组件代码上继续迭代。

**Codex 集成方式：**

```bash
npx ai-elements@latest
npx ai-elements@latest add message
npx shadcn@latest add https://elements.ai-sdk.dev/api/registry/all.json
```

建议让 Codex 先安装最小组件集，而不是一口气引入全部。比如只做聊天页时，先安装 conversation、message、response、prompt-input、code-block，再根据需求补 reasoning 和 tool。

### 3.5 Vercel Chatbot：完整 Next.js AI Chatbot 模板

**项目定位：**

Vercel Chatbot 是基于 Next.js 和 AI SDK 的开源模板，适合快速启动完整 AI Chat 应用。它包含 App Router、RSC、Server Actions、AI SDK hooks、工具调用、多模型 provider、数据库迁移和本地运行流程。

**结构组成：**

- Next.js App Router 应用；
- AI SDK 集成；
- shadcn-ui 风格界面；
- 数据库和迁移；
- 本地 dev 流程；
- Vercel 部署流程。

**适用场景：**

- 想快速获得完整可运行 AI Chat app；
- 需要从成熟模板中学习项目结构；
- 想让 Codex 在模板基础上改成自己的产品；
- 需要部署到 Vercel。

**Codex 集成方式：**

1. 用 GitHub 插件或 `git clone` 获取模板；
2. 让 Codex 阅读目录结构和环境变量要求；
3. 执行 `pnpm install`、`pnpm db:migrate`、`pnpm dev`；
4. 用 Browser 打开 `localhost:3000`；
5. 再让 Codex 按你的产品风格改首页、聊天页、侧边栏、消息状态和登录流程。

### 3.6 LiveKit Agents UI：语音 Agent 的 shadcn 组件

**项目定位：**

Agents UI 是 LiveKit 面向 voice agent frontends 的 shadcn 组件库，构建在 Shadcn 和 AI Elements 之上，提供 session 控制、转写、音频可视化、输入输出控制等组件。

**结构组成：**

- shadcn registry 安装方式；
- `components/agents-ui` 源码目录；
- session provider；
- control bar；
- chat transcript；
- audio visualizer；
- LiveKit realtime primitives。

**适用场景：**

- 语音助手、实时会议 Agent、客服 voice bot；
- 需要音频流、转写、麦克风控制、会话状态；
- 需要 React 19 / Tailwind 4 / shadcn 风格的可定制组件；
- 想把 HeyGen 或语音 Agent 的前端体验做得更产品化。

**Codex 集成方式：**

```bash
npx shadcn@latest init
npx shadcn@latest registry add @agents-ui
npx shadcn@latest add @agents-ui/{component-name}
```

然后让 Codex 根据业务场景挑组件：demo 阶段先上 session provider、control bar、chat transcript、audio visualizer；生产阶段再补断线重连、权限提示、设备选择、空状态和错误状态。

### 3.7 NLUX：跨框架 conversational AI UI

**项目定位：**

NLUX 是 React / JavaScript conversational AI UI library，支持 Next.js、React、Vanilla TypeScript，并提供 LangChain、OpenAI、Hugging Face、Vercel AI 等适配器。

**结构组成：**

- `@nlux/react`：React 组件；
- `@nlux/core`：Vanilla JS 核心；
- LangChain / OpenAI / Hugging Face / nlbridge 适配器；
- themes、markdown、highlighter 扩展；
- `AiChat` UI 和 `useChatAdapter` hook。

**适用场景：**

- 需要轻量聊天 UI；
- 不想依赖外部 UI 库；
- 要兼容 React、Next.js 或普通 JS；
- 需要快速接入多种 LLM backend。

**Codex 集成方式：**

```bash
npx nlux-cli create next my-next-app
npx nlux-cli create react my-react-app
npx nlux-cli create vanilla my-vanilla-app
```

在已有项目中使用时，让 Codex 先判断你的 backend 是 streaming 还是 batch，再创建 adapter。NLUX 更适合“快速可用的 conversational UI”，如果你需要复杂 tool call、approval、agent state，可优先看 assistant-ui、AI Elements 或 CopilotKit。

### 3.8 Chatscope Chat UI Kit React：通用聊天 UI 组件

**项目定位：**

Chatscope Chat UI Kit React 是通用 React chat UI toolkit，提供 `MainContainer`、`ChatContainer`、`MessageList`、`Message`、`MessageInput` 等组件。

**结构组成：**

- `@chatscope/chat-ui-kit-react`；
- `@chatscope/chat-ui-kit-styles`；
- Storybook / demo；
- `@chatscope/use-chat` 状态管理 hook。

**适用场景：**

- 需要常规聊天 UI；
- AI 特性较少，主要是 message list、input、avatar、conversation list；
- 想快速搭建可用聊天界面；
- 不需要复杂 agent protocol 或 generative UI。

**Codex 集成方式：**

```bash
npm install @chatscope/chat-ui-kit-react
npm install @chatscope/chat-ui-kit-styles
```

让 Codex 把它作为基础聊天壳，再接入你的模型 API。它不像 assistant-ui 那样 AI-native，因此 reasoning、tool call、approval、streaming 状态需要你自己补。

## 4. 推荐工作流：让 Codex 真的做出好看的 UI

### 4.1 从需求到高质量页面

1. 用文字说明业务、用户、页面目标、主要操作；
2. 用 Figma 或 imagegen 探索视觉方向；
3. 让 Codex 根据设计稿实现前端；
4. 用 Browser 做桌面和移动端截图验证；
5. 如果是 AI/Agent UI，优先引入 assistant-ui、AI Elements 或 CopilotKit；
6. 把最终页面状态同步回 Figma，沉淀成设计资产。

### 4.2 从页面到演示和传播

1. 页面完成后，用 Browser 或 Chrome 截取关键画面；
2. 用 Canva 生成社媒图、封面、产品说明页；
3. 用 HyperFrames 或 Remotion 做 15-60 秒视频；
4. 需要真人口播时，用 HeyGen；
5. 需要内部汇报时，用 Presentations；
6. 需要科研/机制图时，用 BioRender。

### 4.3 Agent UI 项目的选型建议

| 目标 | 推荐项目 | 原因 |
| --- | --- | --- |
| 产品内嵌 Copilot | CopilotKit | Agent 能读取 UI 状态并触发应用动作 |
| 标准化 Agent 事件流 | AG-UI | 适合跨 agent 框架统一前端协议 |
| 生产级 AI Chat | assistant-ui | streaming、附件、approval、runtime 适配更完整 |
| shadcn + AI SDK | AI Elements | 组件复制到本地，适合 Codex 后续改造 |
| 完整 Chatbot 起步 | Vercel Chatbot | 项目结构完整，适合直接二次开发 |
| 语音 Agent | LiveKit Agents UI | 音频、转写、session 控制天然匹配 |
| 轻量跨框架聊天 | NLUX | React / Next / Vanilla 都能用 |
| 普通聊天壳 | Chatscope | 快速搭建传统 chat UI |

### 4.4 一条组合提示词

```text
我想做一个 AI Agent 产品的前端，请按下面流程执行：

1. 先阅读当前项目结构，判断它是 React、Next.js、Vue 还是其他栈；
2. 根据场景在 assistant-ui、AI Elements、CopilotKit、AG-UI、LiveKit Agents UI 中选择最合适方案；
3. 给出选型理由、需要安装的包、会改哪些文件；
4. 实现一个产品级页面，包含 loading、streaming、error、empty、tool-call、approval、mobile responsive 状态；
5. 用 Browser 打开 localhost 做桌面和移动端视觉验收；
6. 如果 UI 已稳定，再生成一份 Figma/Canva/视频传播资产方案。
```

### 4.5 质量验收清单

- 页面不是 landing page 空壳，首屏就是可用产品界面；
- 按钮、输入框、菜单、tab、toggle、slider、tooltip 等控件符合常见交互习惯；
- 桌面端和移动端都没有文字溢出、元素遮挡、图片空白；
- 颜色不只是单一色相堆叠，层次、对比、状态色清晰；
- Agent UI 覆盖 streaming、tool call、approval、retry、cancel、error、empty、history；
- 视频类产物有明确视觉风格、节奏、字幕、CTA 和渲染验证；
- Figma 类产物优先复用组件、变量、样式，而不是硬画不可维护的图形；
- 外部项目引入后能留在代码库中被 Codex 修改和测试；
- 所有在线工具、第三方模板、素材和开源项目都记录来源与许可证边界。
